# M7 — Caregiver account MFA (TOTP two-factor)

> **Status: FROZEN 2026-05-24.** All six open questions are user-locked
> (see Resolved, below). Coding may begin. Deviations are recorded in
> `docs/milestones/M7.md` or as ADRs — not by editing this spec.

## Goal

A caregiver can protect their account with an authenticator-app second
factor, so a leaked or guessed password alone cannot reach the people
they care for — and can recover access with one-time recovery codes if
their phone is lost, without a support ticket.

## Scope

- **TOTP enrollment** in a new caregiver security surface: show the QR
  code + a copyable manual secret, then require the caregiver to enter a
  current 6-digit code to activate the factor (Supabase
  `auth.mfa.enroll` → `challenge` → `verify`).
- **Recovery codes** — a set generated at enrollment, shown **once**,
  stored only as salted hashes (new `mfa_recovery_codes` table, scoped by
  `caregiver_id` under RLS). Single-use; consumed on use; regenerable
  from settings (regenerating invalidates the old set).
- **MFA challenge step at sign-in** — after `signInWithPassword`
  succeeds, if the caregiver has a verified factor the session is AAL1
  and we redirect to a TOTP challenge page rather than to `/caregiver`.
  A correct code steps the session up to AAL2 and lands the dashboard.
- **AAL2 enforcement on `/caregiver`** — `middleware.ts` gains an AAL
  check beside the existing `!user` redirect: a caregiver who has a
  verified factor but only an AAL1 session is sent to the challenge, not
  the dashboard. Caregivers without a factor are unaffected (AAL1 is
  sufficient for them). Enforced server-side in caregiver data loaders /
  actions as well, not only in middleware.
- **Account recovery (lost device)** — a "use a recovery code" path on
  the challenge screen. A valid code **unenrolls the lost TOTP factor**
  via the privileged admin API (`SUPABASE_SECRET_KEY`), consumes the
  code, and drops the session to AAL1 so the caregiver regains access and
  is immediately prompted to re-enroll. (A recovery code cannot itself
  mint an AAL2 session — Supabase only grants AAL2 from a TOTP verify —
  so this unenroll-then-re-enroll path is the honest mechanism, not a
  shortcut.)
- **Disable MFA** from the security page — requires a fresh TOTP or
  recovery-code verification before unenrolling, and fires an admin
  notification (new `mfa-disabled` kind on M6's `AdminNotificationKind`
  union).
- **Admin runbook entry** in `docs/security/auth-hardening.md` — manual
  factor removal (privileged key / Supabase dashboard) for a caregiver
  who lost both their device and their recovery codes.
- **Enforcement model** — optional opt-in. Caregivers without MFA sign in
  exactly as today; enabling it is a deliberate choice in settings.
- **Patient routes unchanged** — no MFA, no AAL check on `/patient/[id]`.
  Patients have no auth surface (UUID-addressed); this is reaffirmed, not
  modified.
- **Tests:**
  - enrollment / challenge / verify / unenroll server actions (mocked
    Supabase MFA client)
  - recovery-code generation, salted-hash storage, single-use
    consumption, regeneration-invalidates-old
  - middleware AAL2 redirect (verified factor + AAL1 → challenge;
    no factor → through; AAL2 → through)
  - sign-in two-step (password → challenge redirect when factor present;
    straight to `/caregiver` when not)
  - `mfa-disabled` admin-notification template snapshot (new kind)
  - password reset does not drop AAL2 (a caregiver with a factor is
    still challenged after resetting their password)

## Non-goals

- **SMS / phone-number MFA** — out. SIM-swap risk, per-message cost, and
  worse security than TOTP. Not a deferral; not planned.
- **WebAuthn / passkeys / hardware keys** — deferred to a later
  milestone. TOTP is the M7 factor.
- **Session management UI** (active-sessions list, revoke-session) —
  deferred to M8. Supabase refresh-token rotation is the mechanism; the
  UI is its own milestone.
- **New-device login email alerts** — deferred to M8 (fingerprinting +
  geolocation + a new false-positive surface).
- **Password hygiene** (complexity rules, strength meter, HIBP
  pwned-password) — deferred to M8. M6 already argued the pwned-check is
  meaningful only after MFA lands; M7 lands MFA.
- **AAL2 enforcement inside RLS policies** — out for M7. Enforcement is
  at the route/middleware + server-action layer. RLS-level `aal2` gating
  is an M8 hardening item (and is only meaningful once mandatory MFA is
  considered).
- **Patient-facing MFA / any patient auth** — never. Patients reach their
  view by UUID and have no login to protect.
- **Separate/stronger MFA for the `/admin` route** — out. The admin is a
  caregiver whose email matches `ADMIN_EMAIL`; if they enable MFA they
  get it like any caregiver. No distinct admin-only factor.
- **Turnstile on the MFA challenge step** — out. The password step is
  already CAPTCHA-gated (M6); the challenge is post-password and
  rate-limited separately.
- **Rebuilding the retired E2E Turnstile bypass** (M6-prep item) — out of
  M7 unless trivially folded; tracked separately.
- **Mandatory MFA enforcement** — out _if_ the Open Question resolves to
  opt-in; in that case a future "require MFA org-wide" toggle is M8+.

## Acceptance criteria

- [ ] A caregiver with no second factor signs in exactly as today
      (password + Turnstile → `/caregiver`); nothing about their flow
      changes.
- [ ] From the security page, a caregiver can enroll a TOTP factor: scan
      a QR (or copy the secret) into an authenticator app, enter a
      current code, and see the factor become active.
- [ ] On activation, the caregiver is shown a set of one-time recovery
      codes exactly once, with a clear "save these now" prompt; the codes
      are never retrievable in plaintext afterward.
- [ ] After enrollment, signing out and back in requires the password
      **and** a current TOTP code; the dashboard is not reachable at AAL1.
- [ ] Entering a wrong TOTP code shows a calm inline error and does not
      grant access; repeated wrong codes hit Supabase Auth's built-in
      MFA-verify rate limit.
- [ ] A caregiver who lost their device can enter a valid recovery code
      on the challenge screen, which removes the factor, signs them in,
      and prompts re-enrollment. The used code cannot be reused.
- [ ] Recovery codes can be regenerated from the security page; the prior
      set stops working immediately.
- [ ] A password reset (forgot-password flow) does **not** bypass MFA: a
      caregiver with a factor still faces the TOTP challenge after
      resetting their password.
- [ ] Disabling MFA from the security page requires a fresh TOTP or
      recovery-code verification, returns the account to single-factor
      sign-in, and fires an `mfa-disabled` admin notification.
- [ ] The patient view (`/patient/[id]/...`) is reachable exactly as
      before; no MFA prompt, no AAL redirect, ever.
- [ ] `docs/security/auth-hardening.md` documents the lost-device admin
      runbook (service-role factor removal) and the MFA layer is added to
      its layered-defense section.
- [ ] `pnpm typecheck && pnpm lint && pnpm test --run` green; `pnpm build`
      clean.
- [ ] Production smoke: enroll → sign out → sign in with TOTP → recover
      with a code → re-enroll → disable, end-to-end.

## Resolved (spec frozen 2026-05-24)

All six open questions are user-locked; the spec is frozen and M7 coding
can begin. Captured here so a future re-anchor doesn't relitigate them.

1. **Enforcement — opt-in.** Caregivers choose to enable MFA in settings;
   no forced enrollment. Gentler for the older/less-technical audience M6
   flagged. A "require MFA org-wide" toggle is an M8 candidate, not M7.
2. **Recovery — recovery codes + self-serve unenroll.** One-time codes at
   enrollment (salted-hash stored in a new `mfa_recovery_codes` table);
   using one unenrolls the lost factor via the privileged admin API,
   drops the session to AAL1, and prompts re-enrollment. Admin runbook is
   the backstop when codes are also lost. Admin-runbook-only recovery was
   rejected — every lost phone shouldn't become a support ticket.
3. **Privileged key — already present.** The unenroll path uses the
   project's existing `SUPABASE_SECRET_KEY` (new-style secret key, full
   Admin Auth privileges), not a separate `SUPABASE_SERVICE_ROLE_KEY`. A
   privileged-access call-site pattern already exists (see
   `docs/security/rls-audit.md`); reuse it. No new env var needed.
4. **Failed-TOTP limiting — Supabase built-in.** Rely on Supabase Auth's
   MFA-verify rate limits; no custom per-account limiter or `mfa-lockout`
   notification in M7. A 6-digit code under those limits is infeasible to
   brute-force in the verify window.
5. **Disabling MFA — re-verify + notify.** A fresh TOTP/recovery code is
   required to disable, and a new `mfa-disabled` admin notification fires.
   Turning off protection is treated as a security event.
6. **Security-page location — `/caregiver/settings`.** A general settings
   page with a Security section, room to grow, over a security-only route.

## Risks

- **Supabase TOTP has no native recovery codes.** We build our own
  hashed table + a privileged-key (`SUPABASE_SECRET_KEY`) unenroll path.
  Mitigation: isolate privileged-key usage to a single server-only module
  (reusing the existing call-site pattern in `docs/security/rls-audit.md`),
  never ship the key to the client, cover the consume-once logic with
  tests.
- **Caregiver locks themselves out** (enables MFA, loses device + codes).
  The patient is unaffected (UUID access continues), but the caregiver
  can't manage the account. Mitigation: codes shown at enrollment +
  admin runbook backstop; opt-in keeps it from being forced on the
  unprepared.
- **Accessibility.** Authenticator apps are a real hurdle for older
  caregivers (echoes M6's CAPTCHA-accessibility risk). Mitigation:
  opt-in, plain enrollment instructions, printable recovery codes, the
  support-email + admin path.
- **AAL2 enforced at route layer only, not RLS.** A stolen AAL1 cookie
  that somehow bypasses middleware could still read caregiver data via
  server actions. Mitigation: also assert AAL2 in caregiver server
  actions/loaders, not just middleware; flag RLS-level `aal2` as M8.
- **Password reset must not drop the MFA requirement.** Mitigation:
  verify reset lands at AAL1 and still challenges; covered by an
  acceptance criterion + test.
- **Scope creep into session management / passwordless / passkeys.**
  Those are explicit non-goals → M8. If the urge surfaces mid-build, file
  it as M8 prep and keep moving.
