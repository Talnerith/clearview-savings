# M4 — Multi-account, audit log, pending indicator

## Goal

The caregiver gains durable visibility into every change they make and a
second account per patient (savings), and the patient sees their next
scheduled deposit prominently on the home screen — reassuring them that
their money is on the way before it lands.

## Scope

- Patients can have a second account (savings) in addition to checking.
  Caregiver creates it on demand; opening balance is posted as a manual
  adjustment transaction.
- Patient home screen renders each account as its own card. Tapping a
  card opens a per-account view scoped to that account's transactions.
- Caregiver-side flows that previously assumed "the one account"
  (manual transaction, check generation, workbook generation, scheduled
  deposit creation) now take a destination `account_id`. The form
  shows the account picker only when ≥2 accounts exist for that patient.
- Caregiver can transfer money between two accounts belonging to the
  same patient. Atomic: both legs (withdrawal + deposit transactions,
  both balance updates, the audit log row) succeed or none do. Both
  transactions share a `transfer_id` so the audit log can link them.
- Every caregiver-side mutation writes one row to a new `audit_log`
  table with a JSON before/after snapshot.
- Per-patient `/audit` route renders the last 50 actions for that
  patient, filterable by `action_kind` dropdown, with each row
  expandable to show the before/after JSON.
- Patient home screen shows pending scheduled deposits as a prominent
  top-level banner above the account cards. Calm color, friendly date
  phrasing ("Friday", "in 3 days", "Tuesday, March 11"). Each
  scheduled deposit carries its own pending-window length (0–14 days,
  default 5) that the caregiver controls in the scheduled-deposit form.

## Non-goals

- Voiding a deposit code — `deposit_codes.status` stays `unused | used`.
  Reserved as a future `action_kind='code_voided'` value in the audit
  log enum so adding it later is a non-breaking enum extension.
- Audit log retention policy, search, or export — last-50 is enough for
  M4; the full retention/policy conversation is its own milestone.
- Cross-patient transfers — server-side guard enforces same-patient.
- Joint accounts, multiple caregivers per patient, account closure,
  account-type changes.
- Patient-side notification preferences — no notifications exist; this
  milestone does not change that.
- Patient-picked destination at deposit/redeem time. The caregiver
  picks the destination at code-generation time; the patient never
  sees an account picker in the deposit or submit-work flows.
- Pending-banner per-account variant — banner is top-level only for
  M4 (destination account name appears in small text within each
  banner card when ≥2 accounts exist).
- A third account type (money market, CD, etc.). `accounts.type`
  stays `checking | savings`.

## Acceptance criteria

- [ ] Caregiver can create a savings account on the per-patient page
      when one does not exist; cannot create a second savings.
- [ ] Patient home renders all of the patient's accounts as separate
      cards with Available Balance and Recent Transactions per account.
- [ ] Caregiver can transfer money from one of a patient's accounts to
      the other; both transactions appear in the respective accounts'
      recent transactions, with the same `transfer_id`.
- [ ] A transfer that fails mid-operation leaves zero transactions
      written and both account balances unchanged (atomicity test).
- [ ] Every existing caregiver-side mutation (patient create / settings,
      account create / rename, manual transaction, scheduled deposit
      create / edit / pause / delete, check code generate, workbook
      code generate) writes exactly one `audit_log` row with a sensible
      before/after.
- [ ] Caregiver can navigate to `/caregiver/patients/[id]/audit` and
      see the last 50 actions for that patient, filter by `action_kind`,
      and expand a row to view the full before/after JSON.
- [ ] Patient home shows a top-level "Direct Deposit Pending" banner
      above account cards listing every active scheduled deposit whose
      `next_run_at` is within its own `pending_days` window.
- [ ] Pending banner is hidden entirely (no header, no filler text)
      when nothing matches.
- [ ] Caregiver scheduled-deposit form has a "Show as pending N days
      before arrival" field, integer 0–14, default 5.
- [ ] Pending-window logic test: window=5, next_run_at=today+3 days →
      pending; window=5, next_run_at=today+7 days → not pending.
- [ ] No patient-visible string contains the words "audit", "log",
      "transfer", or any other caregiver-side vocabulary leaks.
- [ ] Bank brand on every patient-visible screen routes through
      `getPatientBrand(patient)`.
- [ ] `pnpm typecheck && pnpm lint && pnpm test --run` green.

## Open questions

All resolved before freeze:

- Transfer audit-log shape — **one row** with `action_kind='transfer'`,
  `target_kind='account'`, `target_id=fromAccountId`, both transaction
  IDs in the `after` payload.
- Pending indicator placement — **top-level banner** above account
  cards. Destination account name appears in small type within each
  card when ≥2 accounts exist.
- Transfer scope — **same-patient only**, server-side guard.
- Target-account routing — **caregiver picks at generation time**.
  Patient never sees an account picker in the deposit or submit-work
  flows. Codes generated before M4 (target_account_id is null) fall
  back to the patient's first-created account, matching pre-M4
  behavior.

## Risks

- **Atomic transfer testing** — `db.transaction()` is required for the
  transfer mutation but M2 deliberately avoided it because the test
  backend (pg-proxy + pg-mem) couldn't host transactions. Mitigation:
  verify pg-mem supports BEGIN/COMMIT/ROLLBACK during step 5; if not,
  split the transfer test into a logic-only unit test plus a small
  transactional integration test against a real Postgres instance, or
  a testcontainers-based fixture.
- **Audit log retrofit churn** — every existing caregiver mutation
  needs `logCaregiverAction` wired in. Mitigation: do it as one
  bounded pass (Step 2 of the build plan); fail the typecheck rather
  than ship a partial retrofit.
- **`deposit_codes.target_account_id` nullable column semantics** —
  null means "fall back to first-created account" for pre-M4 codes.
  Risk that a future reader assumes null means "explicitly defer to
  patient choice" and writes confusing UX around it. Mitigation: doc
  the meaning at the column definition and in the redemption path's
  comment.
- **Scope size** — M4 is the largest milestone to date. Mitigation:
  eight build steps with five pause points so the user can verify
  shape after each meaningful chunk rather than at the end.
