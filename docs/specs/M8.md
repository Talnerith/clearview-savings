# M8 — Workbook revamp: gentler skill floor + reward-as-check

> **Status: FROZEN 2026-06-05.** All four open questions are user-locked
> (see Resolved, below). Coding may begin **once M7 is closed** (its
> Step 8 production smoke is the other gate). Deviations are recorded in
> `docs/milestones/M8.md` or as ADRs — not by editing this spec. Change
> #2 (Part B) is backed by ADR 0004, which reverses a documented
> data-model rule.

## Goal

A caregiver can hand their patient a workbook that starts as gently as
kindergarten, and the patient claims the finished-work reward through
the **same "Deposit a Check" flow they already use** — not a second,
separate "Submit Completed Work" code screen.

## Background

Real-use feedback from the user running workbooks with his father
(Alzheimer's), 2026-06-03 (captured in `docs/milestones/M7-progress.md`
§ Deferred and agent memory `workbook-revamp-feedback`):

1. **Grade 2 was still too hard.** The floor needs to drop to
   kindergarten and grade 1. Today `WorkbookGrade = 2 | 3`
   (`lib/workbook-content/types.ts`); banks exist only for grades 2–3
   (`lib/workbook-content/grade-2/`, `grade-3/`).
2. **The typed reward CODE is a barrier.** Today a finished workbook
   prints an 8-char reward code the patient types into a *separate*
   `/patient/[id]/submit-work` screen (`lookupWorkbookCodeAction` →
   `redeemWorkbookCodeAction`). The patient already knows the "Deposit a
   Check" flow; making the reward a check folds two patient mechanisms
   into one.

## Scope

### Part A — lower the skill floor (additive, no ADR)

- Widen `WorkbookGrade` to include **kindergarten and grade 1** (exact
  encoding is Open Question 1).
- Author gentler problem banks for the new levels across all five
  categories (`math-facts`, `word-problems`, `reading-passages`,
  `sequencing`, `simple-logic`) as JSON under
  `lib/workbook-content/grade-{K,1}/`, mirroring the grade-2/3 shape.
- Wire the new banks into the `BANKS` resolver in
  `lib/workbook-content/index.ts`.
- Extend the caregiver generation UI grade picker
  (`app/(caregiver)/caregiver/patients/[id]/workbooks/page.tsx` +
  the `grade` enum in `…/workbooks/actions.ts`) and the `workbookGrade`
  smallint validation in `lib/db/schema.ts` context.
- Extend `bank-sanity.test.ts` / `sampler.test.ts` to cover the new
  levels (every category non-empty, enough problems to fill the page
  shapes without duplication).
- **Naming stays caregiver-facing only** — "kindergarten"/"grade 1"
  never appear in patient-visible UI (the M3 framing rule; the
  transaction label remains a bank-like "Activity Set #N").

### Part B — reward becomes a deposited check (ADR 0004)

- When a caregiver generates a workbook, the finished-work reward is a
  **`kind: "check"` deposit code** carrying the reward amount, rather
  than a `kind: "workbook"` code.
- The workbook PDF presents that **check at the end** (rendering /
  placement is Open Question 2b), with the reward amount and the 8-char
  deposit code printed on it exactly like a standalone check.
- The patient deposits it through the existing
  `app/(patient)/patient/[id]/deposit` "Deposit a Check" flow
  (photo → amount → code → `redeemCodeAction` → `redeemCode`).
- **Remove the patient-facing typed reward-code path:**
  `app/(patient)/patient/[id]/submit-work/**`
  (`lookupWorkbookCodeAction`, `redeemWorkbookCodeAction`, `CodeEntryForm`,
  the `confirm`/`done` pages) and the patient-side
  `deposit_codes.kind = "workbook"` redemption.
- The caregiver answer-key view
  (`…/workbooks/[codeId]/answers`) and workbook PDF
  (`…/workbooks/[codeId]/pdf`) stay — the caregiver still generates and
  reviews workbook content; only the *reward redemption mechanism*
  changes.

## Non-goals

- **AI-generated problem banks.** The `getBank` resolver seam stays
  exactly as documented in `lib/workbook-content/index.ts`; we do not
  build the Anthropic-API generation path here. New levels are
  hand-authored JSON like the existing ones.
- **Changing the check-deposit wizard mechanics.** The photo → amount →
  8-char-code steps in `DepositWizard.tsx` stay. (Whether the *reward*
  check should be lower-friction than a normal check is Open Question
  2a — but reworking the wizard itself is out.)
- **Grades above 3 / a fully configurable difficulty axis.** This
  milestone lowers the floor; it does not add a ceiling.
- **Patient-side gamification** (streaks, badges, reward escalation) —
  still forbidden by CLAUDE.md; the reward stays a single caregiver-set
  amount.
- **Touching auth / MFA** (M7) — orthogonal.
- **A caregiver-configurable reward-vs-code toggle.** Part B is a
  replacement, not an option behind a flag; the typed-code path is
  removed, not hidden.

## Acceptance criteria

- [ ] A caregiver can generate a workbook at the kindergarten and
      grade-1 levels; the printed problems are visibly gentler than
      grade 2.
- [ ] Every category at every level has enough problems to fill the page
      shapes (`PAGE_SHAPES` in `sampler.ts`) without repeating a problem
      within one workbook; `bank-sanity` asserts this.
- [ ] No patient-visible surface contains "kindergarten", "grade",
      "workbook", or other clinical/level vocabulary (forbidden-vocab
      grep over the patient route group stays clean).
- [ ] Generating a workbook produces a reward **check** the patient can
      deposit via "Deposit a Check"; depositing it posts the reward
      amount to the target account exactly once (single-use preserved).
- [ ] The `/patient/[id]/submit-work` route and its actions no longer
      exist; navigating there falls back calmly to the patient home (no
      404, per patient-UX rules).
- [ ] A one-time migration converts every unused `kind = "workbook"`
      deposit code to `kind = "check"`, so rewards printed before M8 stay
      redeemable through the "Deposit a Check" flow (no orphaned-code
      crash, no silently-unredeemable reward).
- [ ] ADR 0004 is committed and the CLAUDE.md data-model note + rule 5
      are updated to match (single-use codes now back checks; the
      workbook reward is delivered *as* a check).
- [ ] `pnpm typecheck && pnpm lint && pnpm test --run` green; `pnpm build`
      clean.

## Resolved (spec frozen 2026-06-05)

All four open questions are user-locked; the spec is frozen. Captured
here so a future re-anchor doesn't relitigate them.

1. **Grade encoding — numeric `0 | 1 | 2 | 3`, K = 0.** The
   `workbookGrade` smallint is unchanged; numeric comparisons keep
   working. K is rendered as "Kindergarten" at the caregiver UI edge
   (and 1 as "Grade 1", etc.). Bank directories follow suit
   (`grade-0/` for K, or `grade-K/` with a 0↔"K" map at the JSON-import
   edge — implementer's call, recorded in `docs/milestones/M8.md`). The
   string-union alternative was rejected as a needless column-type
   change touching every numeric call site.
2. **Reward-check friction — consolidate as-is.** The reward becomes a
   `kind: "check"` code deposited through the existing
   photo → amount → 8-char-code wizard, **unchanged**. The accepted win
   is "one patient mechanism instead of two," not removing code entry.
   A lower-friction reward check (pre-filled/scannable code) would
   reopen the wizard non-goal and is deferred to a possible later
   milestone — not M8.
   - **(2b) Reward-check rendering placement** stays an implementation
     detail for `docs/milestones/M8.md` (final page of the workbook PDF
     vs. a separate printable); it does not gate the freeze.
3. **Backward compatibility — migrate.** A one-time migration converts
   every **unused** `kind = "workbook"` deposit code to `kind = "check"`,
   so pre-M8 rewards stay redeemable through "Deposit a Check." The
   `"workbook"` enum value is **retained** for historical (used) rows;
   no new `workbook` codes are minted. Hard-cutover (dead codes) and
   retain-until-expiry (lingering `submit-work` path) were both
   rejected.
4. **One milestone — both parts ship as M8.** Part A (gentler floor,
   additive, ADR-free) and Part B (reward-as-check, ADR 0004) ship
   together as a single "workbook revamp" milestone. The split into
   M8/M9 was rejected — they are one theme and were requested together.

## Risks

- **Content quality at the K/1 floor.** Too-hard problems defeat the
  point; too-trivial feels patronizing. Mitigation: author against the
  real use case (the user's father), keep the page shapes short, lean on
  the existing seeded-sampler determinism so a caregiver can preview
  before printing.
- **Removing `submit-work` orphans in-flight rewards.** A workbook
  printed before M8 with a `kind="workbook"` code becomes unredeemable
  if the path is deleted outright. Mitigation: resolve Open Question 3
  before coding; cover the chosen path with a test.
- **ADR reverses a CLAUDE.md rule.** Rule 5 and the data-model note name
  workbooks as a code kind. Mitigation: ADR 0004 records the reversal;
  the CLAUDE.md edit is part of the acceptance criteria, not an
  afterthought.
- **Scope creep into wizard redesign.** Open Question 2a could pull the
  deposit wizard into scope. Mitigation: if 2a resolves toward
  lower-friction, file that as its own milestone rather than widening
  M8 mid-build.
- **Sampler page-shape starvation.** Gentler levels may have naturally
  smaller problem pools; `PAGE_SHAPES` counts could exceed a category's
  bank size. Mitigation: `bank-sanity` test asserts each bank ≥ the
  max single-workbook draw for its category.
