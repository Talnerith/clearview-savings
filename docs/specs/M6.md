# M6 — Auth hardening

## Goal

Make Clearview Savings sign-up, sign-in, and password-reset resilient
to external automated abuse — credential stuffing, distributed
brute-force across rotated IPs, and bot-driven sign-up spam — so a
real caregiver's account cannot be plausibly compromised by the
class of attack that hits every public-facing auth surface within
hours of going live.

## Scope

- **Cloudflare Turnstile CAPTCHA** on `/sign-up`, `/sign-in`, and
  `/forgot-password`. Token submitted with the form, verified
  server-side via Supabase Auth's built-in `captchaToken` parameter
  on `signUp`, `signInWithPassword`, and `resetPasswordForEmail`.
  CAPTCHA is bypassed in non-production environments via the same
  `NODE_ENV !== "production"` gate as the existing per-IP limiter,
  and bypassed for E2E tests via a `TURNSTILE_TEST_BYPASS_KEY` env
  var that, when present, accepts a known sentinel token without
  calling Cloudflare. The bypass env var is set only in Vercel
  Preview deployments, never Production.
- **Per-email lockout** layered on top of the existing per-IP
  rate limiter. **5 failed sign-ins to the same email within a
  15-minute sliding window → email locked for 15 minutes**,
  regardless of source IP. Closes the "attacker rotates IPs to
  brute-force one known email" gap the M5 limiter leaves open. The
  legitimate caregiver gets back in by waiting 15 minutes; an
  admin runbook entry covers manual clearance if needed.
- **Admin notification** fires (new `kind: "email-lockout"`
  discriminant on the existing `AdminNotificationKind` union) when
  an email enters lockout. **Silent to the locked-out account
  holder** — no email sent to the address being locked, avoiding
  the attacker-driven-spam vector where repeated lockouts fill a
  victim's inbox. One admin email per lockout event (not one per
  blocked attempt during the lockout window). Same pattern as the
  existing `rate-limit-breach` notification.
- **`docs/security/auth-hardening.md`** — new doc enumerating the
  layered protection model (Vercel Edge → per-IP limiter →
  per-email lockout → CAPTCHA → Supabase internal limits), the
  threat model each layer addresses, and the operational runbook
  for handling lockouts and reviewing rate-limit-breach notifications.
- **Tests:**
  - `lib/rate-limit.test.ts` extended with per-email lockout
    threshold + admin-notification path
  - Mock-Turnstile integration test on the sign-up server action
    asserting the token is forwarded to Supabase Auth
  - Email-template snapshot test for the new `email-lockout` kind
- Update `docs/security/rls-audit.md` with a "see also" pointer to
  `auth-hardening.md` so the two security docs cross-reference.

## Non-goals

- **Vercel Pro / WAF custom firewall rules** — deferred. Staying
  on the Hobby tier for M6; CAPTCHA + Upstash per-IP + per-email
  lockout + Supabase Auth limits + Vercel's free edge DDoS
  mitigation is a sufficient layered defense for a beta-stage
  product. Revisit if real abuse traffic appears in Sentry or
  admin notifications.
- **CAPTCHA on `/reset-password`** (the form where the reset link
  lands) — deferred. The signed Supabase Auth token in the reset
  URL is the security boundary; adding a Turnstile widget to the
  password-change form would friction the legitimate caregiver's
  recovery flow without meaningful security gain. Sign-up,
  sign-in, and forgot-password are protected; the reset-form is
  not.
- **MFA / 2FA (TOTP, SMS, WebAuthn)** — deferred. M6 closes the
  external-threat gaps; MFA is a separate session-security
  milestone with its own enrollment UX, recovery codes, and
  account-recovery flow.
- **Password complexity requirements / strength meter** — deferred.
  Supabase Auth enforces a minimum length (6) by default; we will
  not add a custom regex or per-character class requirement in M6.
- **Pwned-password check (Have I Been Pwned k-anonymity API)** —
  deferred. Useful, but a meaningful add only after MFA lands and
  the marginal value of a stronger password is realized.
- **Session management UI** (active-sessions list, revoke-session
  button on caregiver settings) — deferred. Supabase's refresh-
  token rotation provides the underlying mechanism; the UI to
  expose it is a separate milestone.
- **New-device login email alerts** — deferred. Useful but adds
  fingerprinting, geolocation lookups, and a new false-positive
  surface (legitimate caregivers logging in from a new device get
  spurious "someone signed in" emails); landing it well is a
  milestone of its own.
- **Account lockout self-recovery UI** — deferred. M6 lockouts are
  short (cooldown-based), so the recovery path is "wait for the
  cooldown to expire." Admin can manually clear a stuck lockout
  via direct Upstash key deletion documented in the runbook.
- **Notifying the locked-out account holder by email** — out. A
  per-lockout email to the locked address would create an
  attacker-driven spam vector (lock the same email repeatedly,
  victim's inbox fills with alerts). Only the admin is notified.
- **Per-patient brand override** — cancelled (not deferred) per
  ADR 0002.
- **Marketing emails of any kind** — out indefinitely (carryover
  from M5).
- **Patient-route protection / brute-forcing patient UUIDs** —
  out. Patient UUIDs are 122 bits of entropy; brute-force is
  computationally infeasible regardless of rate limit. Patient
  routes have no auth surface to harden.

## Acceptance criteria

- [ ] `/sign-up` renders a Cloudflare Turnstile widget. Submitting
      the form without solving the widget returns the calm inline
      "please verify and try again" message; the Supabase
      `signUp` call is not reached.
- [ ] `/sign-in` renders a Cloudflare Turnstile widget. Submitting
      without solving fails with the same calm message; Supabase
      `signInWithPassword` is not reached.
- [ ] `/forgot-password` renders a Cloudflare Turnstile widget.
      Submitting without solving fails the same way;
      `resetPasswordForEmail` is not reached.
- [ ] Solved Turnstile tokens are forwarded to Supabase Auth via
      the `captchaToken` option on each auth call, and Supabase
      Auth's "Bot and Abuse Protection → CAPTCHA" toggle is on for
      the project.
- [ ] CAPTCHA is bypassed in non-production environments (same
      `NODE_ENV !== "production"` gate as the existing per-IP
      limiter) so local dev and CI never block on Turnstile.
- [ ] After 5 failed `/sign-in` attempts to the same email within
      15 minutes — from any IP, including 5 different IPs each
      submitting once — the email enters a 15-minute lockout.
      Subsequent sign-in attempts to that email return the calm
      "please wait a moment and try again" message regardless of
      source IP, until the lockout expires.
- [ ] The per-IP limiter from M5 still fires independently — an
      attacker hitting `/sign-in` from one IP 6 times in a minute
      gets the per-IP block on attempt 6 even if no email lockout
      has triggered.
- [ ] Admin receives a Resend email at `ADMIN_EMAIL` when any email
      enters lockout. Subject and body match the new
      `email-lockout` template kind. One email per lockout event
      (not one per blocked attempt during the lockout).
- [ ] Admin can manually clear a stuck lockout via documented
      Upstash key deletion (steps in the runbook section of
      `docs/security/auth-hardening.md`).
- [ ] `docs/security/auth-hardening.md` exists, names every layer
      of the protection stack (Vercel Edge → per-IP limiter →
      per-email lockout → CAPTCHA → Supabase internal limits)
      with what attack class it addresses, includes the
      lockout-clearance runbook, and explicitly marks "Vercel WAF
      custom rules — deferred, Pro tier required."
- [ ] `docs/security/rls-audit.md` cross-references
      `auth-hardening.md`.
- [ ] `lib/rate-limit.test.ts` covers the per-email lockout
      threshold and the admin-notification path; existing per-IP
      tests still pass.
- [ ] CAPTCHA-on-sign-up server-action integration test (mocked
      Turnstile verifier) lands green.
- [ ] `pnpm typecheck && pnpm lint && pnpm test --run` green.
- [ ] Production smoke: sign up + verify email + sign out + sign
      in + forgot-password → reset-password completed end-to-end
      with Turnstile widgets clearing on each step.

## Resolved (spec frozen 2026-05-19)

The original draft of this spec carried six open questions. All
six have user-locked answers as of session close on 2026-05-19;
the spec is frozen and M6 coding can begin. Resolutions captured
here so a future re-anchor doesn't relitigate them.

1. **CAPTCHA provider:** Cloudflare Turnstile. Free, privacy-first
   (no fingerprinting, no third-party cookies), GDPR-clean under
   Cloudflare's privacy policy, supported natively by Supabase
   Auth. hCaptcha rejected — the privacy tier costs money and the
   added vendor wasn't worth the accessibility trade-off.
2. **Per-email lockout:** 5 failed sign-ins per 15-minute sliding
   window per email → 15-minute lockout. Stricter (30-min) and
   looser (10/15→5min) variants rejected; 5/15→15 is the
   credential-stuffing-stopping-without-real-caregiver-pain
   sweet spot.
3. **Vercel Pro / WAF:** Skip Pro, stay on Hobby. Cloudflare
   Turnstile + Upstash per-IP + per-email lockout + Supabase Auth
   limits + Vercel's free edge DDoS is sufficient layered defense
   for a beta-stage product. Revisit if real abuse traffic appears
   in Sentry / admin notifications. Vercel WAF moves to non-goals
   for M6.
4. **Lockout notification:** Silent to the locked-out account
   holder + admin email only (one per lockout event). Matches the
   existing `rate-limit-breach` notification pattern; avoids the
   attacker-driven spam vector where repeated lockouts fill a
   victim's inbox.
5. **CAPTCHA on `/reset-password`:** No. The signed Supabase Auth
   token in the reset URL is the security boundary; adding a
   widget to the password-change form would friction the
   recovery flow without meaningful security gain. The `/reset-
   password` form moves to non-goals.
6. **E2E CAPTCHA bypass:** `TURNSTILE_TEST_BYPASS_KEY` env var.
   Server action checks for this var + a known sentinel token
   value and skips Cloudflare verification when both are present.
   The env var is set only in Vercel Preview deployments, never
   Production. Cloudflare's published always-pass test site keys
   rejected — would render a different widget visually on Preview
   vs Production, slightly confusing for the smoke-tester.

## Risks

- **Turnstile / Cloudflare outage stops sign-up entirely.** If
  Cloudflare's Turnstile API is unreachable, the auth pages
  reject every submission (closed-fail). Mitigation: monitor
  Cloudflare status in `docs/admin-runbook.md`; document the
  manual fallback (admin can temporarily disable the Supabase
  Auth CAPTCHA toggle in the dashboard if Cloudflare has a
  multi-hour outage). The user-visible failure mode is "calm
  inline error, try again later" rather than "site broken" so
  the impact is bounded.
- **Per-email lockout creates a per-caregiver DoS vector.** An
  attacker who knows a caregiver's email can lock that caregiver
  out at will by submitting 5 wrong-password attempts.
  Mitigation: 15-minute cooldown limits the per-attack impact,
  admin gets a `email-lockout` notification, admin can manually
  clear the lockout key via the documented Upstash CLI runbook.
  The attacker gains nothing but harassment value; the real
  caregiver gets back in either by waiting or by escalating to
  admin.
- **CAPTCHA accessibility for older / impaired users.** A
  significant portion of caregivers are themselves older or have
  motor / visual challenges. Mitigation: Turnstile supports audio
  challenge fallback and is designed to be minimally interactive
  for the majority of legitimate users (no "click the
  motorcycles" puzzle). Document the support email path for
  caregivers who cannot clear the challenge.
- **Scope creep into MFA territory.** Once you start auth
  hardening, the natural temptation is to add 2FA "while we're
  here." Strict non-goal — if the urge surfaces mid-build, file
  it as M7 prep and keep moving.
- **Per-email lockout key collision under high concurrency.** The
  Upstash counter increment + threshold check is not atomic in a
  single round-trip; under unlikely race conditions two requests
  could each pass the threshold check and both not trigger the
  lockout-set. Mitigation: use Upstash's atomic `INCR` to read
  the new value and act on that — the lockout-set on the Nth
  increment (not on a separate `GET` + `IF` pattern). Same
  pattern as the existing rate-limit-breach notification at
  `lib/rate-limit.ts:106-112`.
- **Test-backend coverage of per-email lockout.** The new lockout
  logic touches Upstash. Existing rate-limit tests mock
  `@upstash/ratelimit`; the per-email tests can follow the same
  mocking pattern and assert the lockout-set + admin-notification
  sequence without requiring a live Upstash instance in CI.
