# M9 — Lower-friction reward check + real-bank patient UI refresh

> **Status: FROZEN 2026-06-11.** All three open questions are
> user-locked (see Resolved, below). M8 is closed; coding may begin.
> Deviations are recorded in `docs/milestones/M9.md` or as ADRs — not
> by editing this spec.

## Goal

The patient deposits their finished-work reward check from a code they
can actually read — printed oversized, in two easy groups — on a
patient site that looks and feels like the online banking they
remember: full-width brand header, prominent balance band, bank-style
transaction table, at text sizes comfortable for an older reader.

## Background

Two threads, both deferred or raised after M8 closed:

1. **Reward-check friction (M8 Open Question 2a, deferred).** M8
   consolidated the workbook reward into the "Deposit a Check" flow but
   explicitly kept the wizard unchanged (photo → amount → typed 8-char
   code). The spec recorded: "A lower-friction reward check
   (pre-filled/scannable code) would reopen the wizard non-goal and is
   deferred to a possible later milestone." M9 is that milestone — the
   wizard non-goal is deliberately reopened here.
2. **Real-bank look (user request, 2026-06-10).** The user supplied
   screenshots of TD EasyWeb and Coast Capital online banking
   (reviewed in-session; **not** committed to the repo — they contain
   real names and balances). His observation: mainstream bank
   interfaces all share the same skeleton and differ mostly by brand
   color. Clearview Savings should adopt that shared skeleton — it is
   exactly what makes the site feel like "my bank" to a patient who
   spent decades using one — while staying on Clearview's own neutral
   palette and raising text sizes for an older reader.

   Structural conventions distilled from the references:
   - Solid brand-color **header bar** with the logo at the left —
     present on every page, the strongest "this is my bank" cue.
   - **Greeting band** ("Good evening, ⟨name⟩") under the header.
   - A **hero account band** on the account view: account name with
     the balance large and white on the brand color, labeled
     ("Current Balance" / "Available Balance").
   - **Pill/rounded quick-action buttons** directly under the hero
     band.
   - **Transactions as a ruled table** — Date | Description | Amount |
     **Balance** columns, with a running balance per row.
   - Accounts listed as rows in a card: name at left, balance
     right-aligned, generous row height.

## Scope

### Part A — lower-friction reward redemption

- The **reward check's printed deposit code** becomes oversized and
  chunked — two groups of four characters ("ABCD 2345") in very large
  type on the workbook PDF's final check page, so the patient can read
  and copy it without squinting (Resolved 1: mechanism a).
- The deposit wizard's code entry accepts the code typed **with or
  without the group space** and presents the input grouped the same
  way, so what the patient types matches what the check shows. The
  wizard's steps (photo → amount → code) are otherwise unchanged.
- **Plain (standalone) check PDFs are untouched** — the oversized
  treatment is reward-only (Resolved 2).
- Single-use semantics are preserved: the code posts exactly once,
  `status`/`used_at`/`transaction_id` recorded as today.

### Part B — patient UI refresh ("looks like my bank")

- Adopt the real-bank skeleton across the patient route group
  (`app/(patient)/...`): persistent brand header bar, greeting band,
  hero balance band on the account view, quick-action styling for
  "Deposit a Check", bank-style transaction table.
- Add a **running Balance column** to patient transaction lists (the
  hallmark of every reference screenshot).
- Raise patient text sizing: base ≥ 20px (up from 18px), balance
  figures and headings proportionally larger, row heights and touch
  targets generous. The references' sizes are the floor, not the
  target — the user asked for "a little bit bigger" than a real bank.
- Palette stays Clearview-owned and CLAUDE.md-compliant: the accent
  remains the current **deep emerald** (Resolved 3), with neutral
  greys/soft greens around it. WCAG AAA contrast holds everywhere,
  including text on the colored header/hero bands — band shades are
  chosen by contrast math first (the current `emerald-700` is unlikely
  to carry white text at AAA; expect a deeper shade for bands).
- Patient-UX rules all still hold: one primary action per screen, no
  modals/toasts/popovers, no auto-logout, "Tuesday, March 11" dates,
  calm fallbacks, bank-like `<title>` tags, footer disclosure on every
  page.

## Non-goals

- **Imitating any specific bank.** We adopt the *shared* skeleton, not
  TD's or Coast Capital's identity: no TD green, no Coast Capital
  bright blue, no lookalike logo, no copied proprietary names
  ("EasyWeb", "Interac e-Transfer"). CLAUDE.md's impersonation rules
  are unchanged and override any "make it look similar" reading.
- **Sidebar navigation, search, profile menus, promo carousels, or
  product cross-sell panels** from the references — chrome that adds
  cognitive load and dead-end taps for a patient with dementia. The
  patient site stays one primary action per screen.
- **Login/logout chrome on the patient side.** Patients reach the site
  by saved link; adding an auth facade would create a way to get
  locked out.
- **Caregiver dashboard restyle.** The caregiver view keeps its
  current utilitarian look; restyle it later if ever.
- **New patient features** (transfers, bill pay, e-transfer, multiple
  deposit kinds). The skeleton is adopted for what the product already
  does, nothing more.
- **OCR, QR codes, or check-image processing.** The scannable-code
  candidate was rejected at freeze (Resolved 1); we never read amounts
  or codes off a photographed check image; the photo step remains
  accept-and-discard (CLAUDE.md rule 4).
- **Patient-side gamification** — still forbidden; the reward remains
  a single caregiver-set amount.

## Acceptance criteria

- [ ] The reward check's printed deposit code appears in two groups of
      four characters, in type visibly larger than the rest of the
      check body; the standalone (plain) check PDF is unchanged.
- [ ] The deposit wizard accepts the code typed with or without the
      group space; the reward posts to the target account exactly once,
      and a second attempt is calmly refused as today.
- [ ] Every existing check kind — plain checks and unused pre-M9
      codes — still deposits exactly as before.
- [ ] The patient home and account pages show: brand header bar,
      greeting, hero balance band, and a transaction table with Date,
      Description, Amount, and running Balance columns.
- [ ] Patient base font is ≥ 20px; balance figures and headings are
      visibly larger than pre-M9; all text — including white-on-color
      band text — passes WCAG AAA contrast.
- [ ] No patient-visible surface uses TD green, Coast Capital blue, or
      any other major-bank signature palette (design review against the
      CLAUDE.md palette rule).
- [ ] The footer disclosure renders on every patient page, including
      the restyled ones.
- [ ] Forbidden-vocab grep over `app/(patient)` stays clean; page

- [ ] `pnpm typecheck && pnpm lint && pnpm test --run` green;
      `pnpm build` clean.

## Resolved (spec frozen 2026-06-11)

All three open questions user-locked 2026-06-11; captured here so a
future re-anchor doesn't relitigate them.

1. **Reward shortcut mechanism — (a) chunked, oversized code.** The
   code keeps being typed but is printed in two 4-character groups in
   very large type on the reward check; the wizard's code entry
   mirrors the grouping. The QR-scan candidate (b: `BarcodeDetector`,
   bigger win but uneven browser support and an ADR-worthy tension
   with the no-OCR rule) and caregiver pre-arm (c: least patient
   effort but hidden cross-session coupling) were both rejected.
   No ADR needed — (a) reverses no documented rule.
2. **Reward checks only.** Plain check PDFs keep their current
   printed code treatment, exactly as the patient knows them. The
   wizard's code step is one shared screen by necessity, so its
   grouped-entry ergonomics apply to any typed code — but that changes
   no plain-check semantics; the *printed* oversized treatment is what
   "reward only" governs.
3. **Accent palette — keep deep emerald.** The navy alternative was
   rejected; the current emerald identity stays, with band shades
   deepened as needed to carry white text at AAA contrast.

## Risks

- **Bank-realism vs. impersonation line.** A full header-and-hero
  treatment looks much more "bank" than today's cards; a reviewer
  could read a specific bank into it. Mitigation: structural
  conventions only, Clearview-owned palette, explicit design check
  against CLAUDE.md hard rules before merge.
- **AAA contrast on colored bands.** White-on-brand text needs ≥ 7:1;
  many "bank" shades fail — including, likely, the current
  `emerald-700`. Mitigation: pick band shades by contrast math first,
  brand feel second.
- **Quick-action pills creep toward multi-action screens.** The
  references put 4–5 actions under the hero band; we have one.
  Mitigation: the non-goal list caps patient actions at the existing
  ones; "one primary action per screen" is re-checked at review.
- **Running balance correctness.** Per-row balances must reconcile
  with `balance_cents` even with same-timestamp rows. Mitigation:
  compute from an explicit, stable ordering and test it.
- **Restyle regressions in calm-fallback paths.** Error/fallback
  screens are easy to miss in a reskin. Mitigation: acceptance
  criterion covers the footer + vocab grep; smoke the `[...slug]`
  fallback and welcome fallback after restyle.
