# M5 — Production hardening and public launch

## Goal

Clearview Savings goes from a local dev tool to a publicly hosted
application: a caregiver lands on clearviewsavings.com, learns what
the product is, creates an account, verifies their email, and uses
the dashboard — backed by RLS hardening, error tracking, rate
limiting, branded transactional email, and the legal disclosure
required by Canadian regulators on every page.

## Scope

- Brand SVG (`public/branding/clearview-savings-icon.svg`) wired into:
  favicon (via Next.js icon conventions), patient bank header,
  caregiver dashboard header, printed check letterhead, workbook
  covers, transactional email templates.
- A `<FooterDisclosure />` component rendered on every page the app
  serves — marketing, auth, caregiver, patient, legal, error.
- Public marketing landing page at `/` explaining the memory-care
  companion purpose in caregiver-facing language, with **Sign in**
  and **Create account** CTAs (no waitlist).
- Open caregiver sign-up: `/sign-up` is reachable without an invite.
  Supabase Auth's email-verification requirement is the gate before
  reaching `/caregiver`.
- `/check-your-email` post-signup page with calm copy and a resend
  action.
- Resend integration: `lib/email.ts` server helper, React Email
  templates for confirmation and password-reset, Supabase Auth wired
  to Resend via custom SMTP so every auth-generated email flows
  through it. From: `"Clearview Savings" <noreply@clearviewsavings.com>`,
  Reply-To: `support@clearviewsavings.com`.
- Legal / informational pages: `/about`, `/privacy`, `/terms`,
  `/security`.
- Sentry error tracking via `@sentry/nextjs`, with noise filters
  (404s, canceled requests, ResizeObserver, ad-blocker errors) and a
  dev-only `/debug-sentry` route gated by `NODE_ENV !== "production"`.
- Vercel Analytics mounted in the root layout. No custom events.
- Production-grade RLS audit: every table walked, every CRUD policy
  verified, fixes applied as part of M5. Findings in
  `docs/security/rls-audit.md`.
- Cross-tenant integration test: two test caregivers, every table,
  every CRUD, asserts every cross-tenant attempt fails.
- Admin notifications via Resend to `ADMIN_EMAIL`. Notification types
  bounded to 4 (see Open Questions).
- Rate limiting via Upstash on `/sign-up`, `/sign-in`,
  `/forgot-password`. 5 attempts/min/IP, calm "please wait a moment
  and try again" message. Skipped in non-production.
- Vercel deployment at apex `clearviewsavings.com` and `www`. All
  required env vars documented in `docs/deployment.md`.
- Documentation: `docs/deployment.md`, `docs/security/rls-audit.md`,
  `docs/admin-runbook.md`, launch-ready `README.md`.
- Tests: RLS cross-tenant integration test, email-template snapshot
  tests, rate-limit threshold tests, sign-up → verify → dashboard
  E2E.

## Non-goals

- **Invite-only waitlist** — original M5 brief specified one;
  pivoted to open sign-up with email verification. No
  `waitlist_signups` table, no `/admin/waitlist` route, no invite
  tokens.
- **Per-patient brand override** — Milestone 6.
- **Multiple admins / role-based admin** — single `ADMIN_EMAIL` env
  var, exact string match.
- **Multi-caregiver per patient** — out indefinitely.
- **Real payment integration** — out indefinitely; the simulation
  premise forbids it.
- **French / multi-language** — out indefinitely.
- **Account deletion self-serve** — manual via `support@` for now;
  documented in admin runbook.
- **Audit log retention / export / search** — deferred (M4 punted,
  M5 punts).
- **Marketing emails of any kind** — out indefinitely; transactional
  only.
- **Cookie consent banner** — Vercel Analytics is cookieless;
  Sentry's tracing is configured cookieless; no banner needed.
- **Patient-side responsive polish beyond what exists** — out of
  scope; today's responsive layout is acceptable for M5.

## Acceptance criteria

- [ ] `clearviewsavings.com` resolves to a public marketing page with
      branded header, the Clearview Savings SVG icon, footer
      disclosure, and Sign in / Create account CTAs.
- [ ] Anonymous visitor can navigate to `/about`, `/privacy`,
      `/terms`, `/security`; each renders with the footer disclosure
      and explicit "Clearview Savings is a memory-care companion
      application, not a real financial institution" language.
- [ ] A caregiver can submit `/sign-up` without an invite token. On
      submit, an unconfirmed `auth.users` row exists and a
      Clearview-Savings-branded confirmation email is delivered via
      Resend to the typed address.
- [ ] `/check-your-email` page renders after sign-up with calm copy
      and a resend-confirmation action that re-issues the email.
- [ ] Clicking the link in the confirmation email verifies the email,
      signs the caregiver in, and lands them at `/caregiver`.
- [ ] A caregiver whose email is not yet confirmed cannot reach
      `/caregiver`; they bounce back to a calm pre-confirmation
      screen.
- [ ] Password-reset flow sends a Clearview-Savings-branded email via
      Resend; the reset link lands at a working `/reset-password`
      page.
- [ ] All Supabase-Auth-generated emails (confirmation, reset, any
      future ones) flow through Resend with `From: "Clearview
      Savings" <noreply@clearviewsavings.com>`, `Reply-To:
      support@clearviewsavings.com`, and the branded template.
- [ ] `<FooterDisclosure />` renders verbatim "Clearview Savings is a
      memory-care companion application. Learn more." with "Learn
      more" linking to `/about`, on every page the app serves
      including patient routes.
- [ ] Patient bank header, caregiver dashboard header, printed check
      letterhead, and workbook covers all show the Clearview Savings
      SVG icon.
- [ ] A dev-mode hit to `/debug-sentry` triggers a captured error in
      Sentry; a production-mode hit to `/debug-sentry` returns 404.
- [ ] Vercel Analytics is mounted in the root layout (analytics
      endpoint requested on page load).
- [ ] `docs/security/rls-audit.md` enumerates every table, every CRUD
      policy, and the audit's verdict per policy.
- [ ] Cross-tenant integration test creates two caregivers A and B,
      then asserts every cross-tenant SELECT / INSERT / UPDATE /
      DELETE attempt by A against B's data fails.
- [ ] Admin receives an email at `ADMIN_EMAIL` when a new caregiver
      verifies their email and lands a caregivers row.
- [ ] `/sign-up`, `/sign-in`, `/forgot-password` return a calm
      "please wait a moment and try again" message after 5
      attempts/min/IP (production only).
- [ ] Application is deployed to Vercel and resolves at both apex
      `clearviewsavings.com` and `www.clearviewsavings.com`.
- [ ] `pnpm build` runs locally without errors.
- [ ] `docs/deployment.md` lists every required env var with
      acquisition steps.
- [ ] Patient-visible strings audit passes — same forbidden
      vocabulary list as M4 (`audit`, `log`, `transfer`,
      `workbook`, `worksheet`, `grade`, `exercise`, `activity
      book`, `Alzheimer`, `dementia`, `simulated`, `fake`, `demo`,
      `therapeutic`). The footer disclosure's "memory-care companion
      application" wording is the single approved exception, fixed by
      CLAUDE.md.
- [ ] No B4A reference exists in the repo outside
      `docs/milestones/` and the single explanatory comment in
      `scripts/apply-migration.ts`.
- [ ] `pnpm typecheck && pnpm lint && pnpm test --run` green.

## Open questions

These need answers **before any M5 code is written**.

1. **Email-template icon embedding strategy.** React Email templates
   can carry the brand icon three ways: inline `<svg>` (small,
   stripped by Gmail and Outlook), inline base64-encoded PNG (works
   in most clients, bloats body, requires a PNG rasterization of
   the SVG), or external `<img src="https://clearviewsavings.com/
   branding/clearview-savings-icon.svg">` (simplest, requires the
   site to be live and the asset to be cached well). **Proposing
   external `<img>` referencing the asset at clearviewsavings.com,
   with the brand name in a typographic header as the always-visible
   fallback.** Need: confirm or pick another option.
2. **Sentry sample rates and replay.** Proposing
   `tracesSampleRate: 0.1`, errors at `1.0`, **no session replay**
   (replays could capture patient-visible UI; privacy posture is
   "minimum data captured to operate"). Need: confirm rates and the
   no-replay stance.
3. **Sign-up form fields.** Currently `/sign-up` takes email +
   password. **Proposing leaving it as email + password only** — a
   caregiver display name is collected at first-patient-add, where
   it's already part of the form. Need: confirm or add a display
   name field to sign-up.
4. **Admin gating shape.** Proposing exact string match between
   `auth.user.email` and `process.env.ADMIN_EMAIL` on every
   `/admin/*` request. Single admin only. Need: confirm shape (no
   comma-separated list, no DB role).
5. **Admin notification list.** Proposing four notification kinds,
   all Resend → `ADMIN_EMAIL`: (a) new caregiver verified email and
   created a caregivers row, (b) repeated auth failures from a
   single IP exceeded rate limit threshold, (c) any Sentry
   `level=fatal` event, (d) a daily caregiver-count digest at
   09:00 UTC. Need: confirm or trim/add.
6. **E2E framework.** Proposing Playwright. Trade-off vs.
   `vitest + msw` (lighter, no browser, can't fully exercise email
   click-through). Need: confirm Playwright is acceptable, or pick
   the lighter alternative.

## Risks

- **Email deliverability is binary in production.** Resend domain is
  set up but the first real send through the Supabase-Auth-custom-
  SMTP chain is the truth. Mitigation: dev send with a verified
  Resend test address before the user flips the Supabase SMTP toggle
  in production.
- **Supabase custom-SMTP toggle is dashboard-only.** User must flip
  it; spec calls out the pause; plan doc carries the exact path.
  Mitigation: explicit pause point in the implementation plan.
- **RLS audit may find real bugs.** That's the point. Fixes are
  in-scope for M5 (not deferred). Mitigation: audit early in the
  milestone so any large fix has runway.
- **Brand swap (Step 1) touches many files.** Mitigation: typecheck +
  build after; grep for old patterns; visual smoke of each surface.
- **`/debug-sentry` accidentally shipped to production.** Mitigation:
  explicit `NODE_ENV` gate + a route-level smoke test that asserts
  it 404s when `NODE_ENV === "production"`.
- **Footer wording carries "memory-care companion."** That phrase is
  borderline against the patient-vocab rule (M4 forbade adjacent
  terms). The CLAUDE.md footer-rule sharpening explicitly carves out
  this exact wording on every page including patient routes as the
  required Canadian disclosure. Mitigation: the patient-vocab audit
  in Acceptance criteria names this as the single approved
  exception, so a future audit doesn't flag it as a regression.
- **DNS propagation timing.** Apex + www records to Vercel can take
  minutes to hours. Mitigation: schedule late in the milestone so
  dev iteration isn't blocked; document the records in
  `docs/deployment.md` ahead of time.
- **Test-backend transaction limits from M4 carry forward.** New
  M5 tests (rate limit, RLS) must respect pg-mem / pg-proxy
  limitations. Mitigation: documented in `docs/gotchas.md` already;
  no new test work fights the backend.
- **Scope size.** M5 is the largest milestone yet — explicitly
  expected to span multiple sessions. Mitigation: build steps are
  pre-numbered with a pause after each, and a re-anchor protocol
  lives in `docs/milestones/M5.md` for fresh sessions.
