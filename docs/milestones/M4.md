# Milestone 4 — Multi-account, audit log, pending indicator

Building on `m3-complete`. M4 is the largest milestone to date and
splits roughly in thirds: a caregiver audit log retrofit, a second
account per patient with everything that needs to flow from that, and
a calmer prominent rendering of the existing M1 "Direct Deposit
Pending" panel.

The frozen pre-flight contract is `docs/specs/M4.md`; this file
records the build plan and any implementation-time decisions that
deviate from or extend that contract.

## Scope and decisions

### Data model — additive only

One migration (`0003_*.sql`) adds:

- `audit_log` table: `id`, `caregiver_id` (FK caregivers, cascade),
  `patient_id` (FK patients, set null, nullable for caregiver-level
  actions), `action_kind` (enum), `target_kind` (enum), `target_id`
  (uuid, nullable for the few caregiver-level actions that have no
  primary entity), `before` jsonb (null on create), `after` jsonb
  (null on delete), `note` text (nullable), `created_at` timestamptz
  default now. Indexed on `(caregiver_id, patient_id, created_at desc)`
  for the per-patient view's last-50 query. RLS: select / insert
  where `caregiver_id = current_caregiver_id()`, same helper M1 uses.
- `audit_action_kind` enum: `patient_created`, `patient_settings_updated`,
  `account_created`, `account_renamed`, `transaction_created`,
  `scheduled_deposit_created`, `scheduled_deposit_updated`,
  `scheduled_deposit_paused`, `scheduled_deposit_deleted`,
  `check_code_generated`, `workbook_code_generated`, `transfer_made`,
  `code_voided` (reserved, not emitted in M4 — spec acceptance
  criterion forbids adding the void feature this milestone, but
  reserving the enum value means the future feature is a
  non-breaking addition).
- `audit_target_kind` enum: `patient`, `account`, `transaction`,
  `scheduled_deposit`, `deposit_code`.
- `transactions.transfer_id` uuid nullable. Both legs of a transfer
  share the same value. No FK — the value is a self-grouping id, not
  a reference. Indexed for "show both legs together" queries.
- `scheduled_deposits.pending_days` smallint, default 5, with a CHECK
  constraint `0 <= pending_days <= 14`. Pre-M4 rows backfill to 5.
- `deposit_codes.target_account_id` uuid nullable, FK accounts(set
  null on delete). Null = fall back to the patient's first-created
  account, preserving M2/M3 semantics for any pre-existing rows.
  Column doc-comment names this convention explicitly.

### Audit log helper — single call site per mutation

`lib/audit-log.ts` exports one function:

```ts
async function logCaregiverAction(db, args: {
  caregiverId: string;
  patientId: string | null;
  actionKind: AuditActionKind;
  targetKind: AuditTargetKind;
  targetId: string | null;
  before?: unknown;
  after?: unknown;
  note?: string | null;
}): Promise<void>
```

Every caregiver-side server action calls this exactly once, after
the underlying mutation succeeds. If the mutation throws, the log
row is not written — the action did not happen.

We deliberately do NOT wrap the mutation + log-write in
`db.transaction()`. The atomic invariant we care about is "the
mutation happened iff the log row exists", not the other direction:
a successful mutation followed by a failed log write is a real
inconsistency we tolerate (the change happened but isn't audited),
caught at the application layer by ensuring the log write is the
last thing in the action. The reverse — a logged action that didn't
happen — is what we cannot tolerate, and the order-of-operations
gives us that for free.

Exception: the transfer flow does wrap both legs and the log write
in one `db.transaction()` because the transfer's atomicity is the
feature.

### Audit retrofit pass (Step 2)

Touches every existing caregiver mutation. The mutations to retrofit
live in:

- `app/(caregiver)/caregiver/patients/actions.ts` — patient create
  (and any settings-update path; verify what exists).
- `app/(caregiver)/caregiver/patients/[id]/actions.ts` — account
  create / rename (if rename exists), manual transaction post,
  scheduled deposit create / edit / pause / delete.
- `app/(caregiver)/caregiver/patients/[id]/checks/actions.ts` —
  check code generate.
- `app/(caregiver)/caregiver/patients/[id]/workbooks/actions.ts` —
  workbook code generate.

Step 2 is a refactor pass — no new caregiver UI changes. The audit
viewer (`/caregiver/patients/[id]/audit`) is built in step 3.

### Target-account routing in checks & workbooks

The caregiver form gets an account picker (dropdown — caregiver
side, standard density is fine) ONLY when the patient has ≥2
accounts. With 1 account the form omits the field and the action
sets `target_account_id` to that single account's id explicitly
(safer than leaving it null and relying on the fallback).

Redemption (`lib/deposit-codes.ts redeemCode`) now reads
`target_account_id` from the code row. If null, it falls back to
the existing "first account by created_at asc" path. The redeem
test gets a new case asserting target_account_id is respected.

### Patient home — account cards

The home page query already pulls all accounts for the patient and
filters transactions/scheduled deposits by `accountIds`. M4
re-renders the existing data: each account becomes its own card
with its own Available Balance and Recent Transactions section.
Visual treatment unchanged from M1 — same emerald accent, same
font sizes — to preserve patient familiarity.

Per-account route: `/patient/[id]/accounts/[accountId]/page.tsx`.
Server component, ownership verification at lookup (account belongs
to a patient whose id matches the URL param), full transaction
history for that account scoped to the last N. Same calm
WelcomeFallback degradation pattern as the rest of the patient
routes (per M3 hardening commit).

### Transfer feature (Step 5)

UI: `/caregiver/patients/[id]/transfer/page.tsx` or in-context
button on the per-patient page (decide during implementation;
in-context is simpler if the per-patient page has room).

Mutation: `transferAction(formData)` in
`/caregiver/patients/[id]/transfers/actions.ts`. Validates both
account IDs resolve to the same patient_id owned by the current
caregiver. Wraps the following in `db.transaction()`:

1. Insert two rows in `transactions` (one negative on `from_account`,
   one positive on `to_account`, both carrying the new `transfer_id`
   uuid).
2. Update both account balances.
3. Insert the audit log row.

If the test backend (pg-mem) does not support transactions, the
fallback is a transactional integration test against a real
Postgres instance (testcontainers or local dev DB); document the
choice in this file at Step 5 time.

### Pending banner (Step 6)

Logic lives next to the existing `materializeScheduledDeposits` at
`lib/scheduled-deposits/`. Add a sibling `getPendingDeposits(db,
patientId, now)` returning `Array<{ scheduledDepositId, label,
amountCents, nextRunAt, accountId, accountName }>` filtered by
`active = true AND next_run_at <= now + pending_days`.

Banner component at `app/(patient)/patient/[id]/PendingBanner.tsx`
(server component is fine — no interactivity). Renders one panel
per pending item, ordered by `nextRunAt asc`. Each panel: label,
amount in patient's currency, arrival phrasing.

Date phrasing helper at `lib/format-arrival.ts` returning:
- `"in 3 days"` for `1 < daysAway < 7`
- weekday name (e.g. `"Friday"`) for `7 <= daysAway < 14`
- long format `"Tuesday, March 11"` for `daysAway >= 14` (won't
  trigger under default 5-day window but a 14-day window can hit
  this branch)
- `"tomorrow"` for `daysAway === 1`
- `"today"` for `daysAway === 0`

Account-name suffix only when patient has ≥2 accounts ("Pension of
$1,800 will arrive on Friday in your Checking account.").

### Patient vocabulary checks (Step 8)

Tightened from M3's pass — new terms now forbidden in
`app/(patient)/**`:

- `\baudit\b`, `\blog\b`, `\btransfer\b` (caregiver vocabulary)
- existing M3 terms: `\bworkbook\b`, `\bworksheet\b`, `\bgrade\b`,
  `\bexercise\b`, `\bactivity book\b`

Patient-visible terms allowed: "Pending", "Direct Deposit Pending",
"Available Balance", "Recent Transactions", "Checking", "Savings",
account names the caregiver chose.

### Tests (Step 7)

- `lib/transfers/transfer.test.ts` — transfer atomicity: success
  path writes 2 transactions + 2 balance updates + 1 audit row;
  forced-failure path writes none. Forced failure simulated by
  triggering a unique-constraint violation on the second insert
  (insert with a `transfer_id` that pre-exists if we add a unique
  partial index, or simulate at the test level).
- `lib/audit-log.test.ts` — table-driven coverage test: for each
  representative mutation in each category, run it through a
  realistic call site, assert exactly one audit row with the
  expected `action_kind`, sensible `before`/`after` payload, and
  correct `caregiver_id` / `patient_id` scoping.
- `lib/scheduled-deposits/pending.test.ts` — pending logic: matrix
  over (next_run_at offsets, pending_days values) verifying inclusion
  rules; ensure inactive scheduled deposits never appear.

### What stays in M3 shape

- The deposit / submit-work flows on the patient side are
  unchanged. M4 does not add a patient-side account picker (per the
  resolved D question — caregiver-only routing).
- The deposit/submit confirmation pages, the done pages, and the
  WelcomeFallback degradation pattern are untouched.
- The pool-cached postgres client and the patient-route try/catch
  defensive layer (from the post-M3 hardening commit) stay as-is.
