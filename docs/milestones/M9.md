# Milestone 9 — Lower-friction reward check + real-bank patient UI

The frozen pre-flight contract is `docs/specs/M9.md` (frozen
2026-06-11); this file is the implementation plan and records
implementation-time decisions that deviate from or extend that
contract. No ADR backs M9 — the chosen reward mechanism (chunked
oversized code) reverses no documented rule, and the UI refresh
operates entirely inside CLAUDE.md's existing branding and patient-UX
rules.

M9 has two parts:

- **Part A — lower-friction reward redemption.** The reward check's
  deposit code prints oversized in two 4-character groups; the wizard's
  code entry mirrors the grouping. Reward checks only; plain check PDFs
  untouched.
- **Part B — real-bank patient UI refresh.** The patient route group
  adopts the mainstream-bank skeleton (brand header bar, greeting band,
  hero balance band, transaction table with running Balance column) in
  Clearview's own deep-emerald palette, base font raised to 20px.

## Gate

None — M8 is closed (2026-06-06). Coding may begin on user "go."

## Resolved (from the spec, locked at freeze 2026-06-11)

Full text in `docs/specs/M9.md` § Resolved:

1. **Mechanism (a) — chunked, oversized code.** Typing stays; the
   reward check prints the code as "ABCD 2345" in very large type. QR
   scan and caregiver pre-arm rejected. No ADR needed.
2. **Reward checks only.** The oversized *printed* treatment applies
   only to the workbook PDF's final check page. The wizard's single
   shared code step gains grouped-entry ergonomics for any typed code
   (no plain-check semantics change).
3. **Palette — keep deep emerald.** Navy rejected. Band shades deepen
   as needed to carry white text at WCAG AAA (≥ 7:1).

## Deviations from the frozen spec

### Printed checks stay unchanged — Step 1 reverted (2026-06-11)

At the Step 2 pause the user corrected a misreading that had shaped
Part A: his "print needs to be larger" feedback was about the **patient
web UI** (the account view), not the printed artifacts — "the checks
were actually great, needed no change." The spec's oversized-chunked
*printed* code (Resolved 1, acceptance criterion 1) was therefore
built on a wrong premise and is dropped:

- Step 1 (CheckPage `codeEmphasis`, taller check box, chunked print)
  is **reverted**; both PDFs print exactly as they did at M8 close.
- Step 2 **stands**: the wizard's grouped, larger code entry and the
  whitespace-tolerant `redeemCodeAction` are entry-side ergonomics the
  user did not object to. Note the print shows `ABCD2345` unspaced
  while the entry field displays `ABCD 2345` — the action accepts
  both, so nothing breaks; spec criteria 2–3 still hold.
- Spec acceptance criterion 1 is recorded here as **waived** (its
  premise was wrong), per the frozen-spec lifecycle. Part A's real
  payload was always Part B's text sizing, which proceeds unchanged.

## Implementation-time decisions recorded here

### One grouped input, not two boxes (2026-06-11)

The spec's mechanism text says "two 4-character boxes printed … on the
check" — that governs the *print*. For *entry*, the wizard keeps a
**single input** that auto-inserts the group space after the 4th
character (display "ABCD 2345", value normalized to 8 chars on
submit). Two separate inputs were rejected: focus jumping between
boxes is its own friction for a patient with dementia, and a single
target preserves "one primary action" simplicity. `normalizeCode`'s
`ALPHABET_FILTER` already strips spaces, so pasted/typed codes with or
without the space normalize identically.

### `CheckPage` gains a `codeEmphasis` prop (2026-06-11)

`lib/check-pdf.tsx` exports the shared `CheckPage` (used by both the
standalone check PDF and the workbook PDF's final page since M8).
Rather than forking the layout, `CheckPage` takes
`codeEmphasis?: "standard" | "large"` (default `"standard"`).
`lib/workbook-pdf.tsx` passes `"large"`: the code renders chunked
("ABCD 2345") at display size — the dominant element of the code row —
while the standalone check keeps today's `codeText` treatment
(`check-pdf.tsx:204`, printed at `:327`). One component, one switch,
spec's "plain checks untouched" holds by default.

### Patient header bar lives in a new `[id]/layout.tsx` (2026-06-11)

The brand header bar must appear on every patient page, which today
each render their own `<Brandmark>` header ad hoc. A new
`app/(patient)/patient/[id]/layout.tsx` renders the persistent header
band (brand at left, white-on-emerald, linking home) above
`{children}`; per-page headers are removed. The route-group layout
(`app/(patient)/layout.tsx`) keeps owning the base typography because
it has no access to `params`. The layout's patient lookup is
best-effort (brand falls back to `getPatientBrand()` on any failure)
so the calm-fallback paths never lose the header.

### Band shade chosen by contrast math (2026-06-11)

White text on `emerald-700` (today's button color) is ~4.5:1 — AA, not
AAA. Header/hero bands therefore use a deeper shade (start from
`emerald-900`, verify ≥ 7:1 with a contrast checker before committing;
record the final token here). Buttons keep `emerald-700/800` where
their text size qualifies for AAA large-text thresholds — verify, don't
assume.

### Round-2 review: two spec non-goals reversed (2026-06-11)

The user's round-2 verdict reversed two frozen non-goals, both recorded
here per the frozen-spec lifecycle:

1. **Busy patient chrome (was: non-goal "sidebar navigation … promo
   carousels").** The field test inverted the premise: the patient
   senses the site is *not real* because it is too sparse. Decorative,
   non-interactive bank chrome added — nav strip (`PatientNav.tsx`) and
   info rail (`InfoRail.tsx`) — per **ADR 0006**, which carries the
   guardrails (truthful panel text, aria-hidden fake controls, no
   real-bank product names, no deposit-insurance claims).
2. **Site-wide restyle (was: non-goal "caregiver dashboard
   restyle").** The user wants one product, one look: marketing, auth,
   and caregiver surfaces synced to the M9 design language. Done via
   shared tokens, not per-page rework: `--primary` rethemed to
   emerald-700 (was shadcn near-black), shared `Button` gains
   cursor/hover/active press states, headers scaled to the patient
   chrome (max-w-6xl, Brandmark lg), landing hero on the emerald-900
   band, caregiver `loading.tsx`. Caregiver density unchanged
   (CLAUDE.md).

Also in round 2 (no spec text reversed):

- **Currency is no longer configurable or visible.** The caregiver
  Currency field is removed; `settings.currency` is derived from the
  locale's region on save (`lib/locale-currency.ts` + test), and every
  money formatter uses `currencyDisplay: "narrowSymbol"` so amounts
  always read "$1,234.56", never "US$1,234.56" — a bank shows amounts
  in its own country's currency with no qualifier. Stored settings
  shape unchanged; no migration. (Printed checks render identically
  for matching locale/currency pairs; mismatched pairs lose only the
  "US$" prefix.)
- **Full display name in the patient greeting** (round 1, recorded
  below) stands.

### Visual-review round-1 adjustments (2026-06-11, pause 2 of 3)

The user's verdict on the Step 6 review: still reads like a mobile app.
Three adjustments landed before Step 7:

1. **Desktop-dedicated patient UI — ADR 0005.** The patient site is a
   PC experience (mobile comes later as a separate app). Containers
   widened max-w-3xl → max-w-6xl (header, home, account), max-w-2xl →
   max-w-3xl (wizard/done); greeting and hero bands compose left/right
   like the reference desktops; the transaction table's Date column no
   longer collapses on small screens; "Tap to take a photo" → "Click to
   add a photo"; `capture="environment"` dropped from the photo input;
   header Brandmark bumped to `lg`.
2. **Reactive controls.** Every button/link gets hover + `active:`
   pressed states (and explicit `cursor-pointer` — Tailwind v4 preflight
   no longer sets it on buttons); primary buttons compress slightly on
   press (`active:scale-[0.98]`); the wizard's submit shows a spinner
   next to "Working…"; a new `patient/[id]/loading.tsx` gives instant
   "One moment…" feedback (spinner, calm copy, header band persists)
   during server-rendered navigations.
3. **Full name in the greeting.** "Good evening, ⟨full display name⟩."
   — the `firstName()` helper is removed; the patient's own full name
   is a stronger "my bank" recognition cue.

### Running balance computed backwards from `balance_cents` (2026-06-11)

Per-row Balance = account's current `balance_cents` minus the sum of
amounts of all *newer* transactions, walked over a stable ordering
(`posted_at` desc, `id` as deterministic tiebreak for same-timestamp
rows). Computing backwards from the authoritative current balance means
the newest row always reconciles with the hero band even when older
rows are cut off by the page's TX limit. Lives in a pure helper
(`lib/running-balance.ts`) with co-located tests; pages call it after
their existing queries.
