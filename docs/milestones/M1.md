# Milestone 1 — Caregiver auth, patient view, scheduled deposits

Tagged at `m1-complete` (commit `f08d3d2`).

This milestone establishes the foundation: a real Supabase project, an
authenticated caregiver web UI, an unauthenticated patient web UI, and the
materialize-on-load pipeline that makes scheduled deposits feel automatic
without any cron infrastructure.

## Deviations from the original plan

Things that turned out differently than the brief implied. Mostly forced
by Windows / Supabase / Next.js realities, none of them load-bearing.

### Step 1 — Scaffold

- **Bumped Next.js to 15.5.18** (from the 15.1.x line) to clear
  CVE-2025-66478. No API changes affected our code.
- **Removed `experimental.typedRoutes`** — Next now warns that the option
  has been moved out of `experimental`. We use the default behavior.
- **Used the new-style Supabase keys** (`sb_publishable_*` /
  `sb_secret_*`) under env vars `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` and
  `SUPABASE_SECRET_KEY` instead of the legacy `*_ANON_KEY` /
  `*_SERVICE_ROLE_KEY` names. This was an explicit user request and is
  applied across every config file, helper, and seed script.

### Step 2 — Schema + RLS

- **`drizzle-kit push` hung silently on Windows** (interactive prompt that
  doesn't render under `pnpm`). Worked around with a custom
  `scripts/apply-migration.ts` (mapped to `pnpm db:migrate`) that reads
  `drizzle/*.sql` in order and applies via `postgres.js` `.unsafe()`. Use
  `pnpm db:migrate` going forward, not `db:push`.
- **RLS policies live in `supabase/policies.sql`**, applied via
  `pnpm db:policies` (also a custom tsx script). Policies are written to
  be idempotent: each is `drop policy if exists ... ; create policy ...`.
  Re-applying is safe.
- **`DATABASE_URL` must be the Supabase _session_ pooler URL** (port 5432),
  not the transaction pooler (6543). Drizzle migrations and Drizzle from
  Server Components use prepared statements, which the transaction pooler
  rejects. Special characters in the password (e.g. `#`) must be URL-
  encoded.
- The partial unique index
  `transactions_scheduled_occurrence_uniq (scheduled_deposit_id,
  scheduled_occurrence_date) WHERE scheduled_deposit_id IS NOT NULL` is the
  load-bearing piece for `materializeScheduledDeposits` idempotency. Don't
  remove it without replacing the guarantee.

### Step 3 — Caregiver auth

- Sign-up uses Supabase's email confirmation flow. The `Site URL` and
  `Redirect URLs` must be configured in the Supabase dashboard
  (`Authentication → URL Configuration`) before the confirmation email
  link will land back at `/auth/callback`. For local dev:
  - Site URL: `http://localhost:3000`
  - Redirect URLs: include `http://localhost:3000/auth/callback`
- `getCurrentCaregiver()` auto-creates the `caregivers` row on first
  sign-in if one doesn't exist. The seed script also creates it
  explicitly because it needs the row before inserting a patient.

### Step 4 — Caregiver dashboard

- **`amountCents` is a _signed_ delta to the account balance.** Deposits
  store positive cents, withdrawals/fees store negative cents, and
  manual adjustments are signed per a `direction` field on the form.
  The materializer follows the same convention. Patient view derives
  the colour and sign of each transaction from the cents value, not
  from the `kind`.
- All balance changes happen inside `db.transaction(...)` alongside the
  inserted row. Don't add a code path that mutates `accounts.balance_cents`
  without inserting a `transactions` row in the same transaction.
- The brief mentioned an audit log of every adjustment. **No audit log
  table was added** in M1 — the schema doesn't include one. Deferred to a
  later milestone (see TODOs).
- Pause/resume of a scheduled deposit is implemented as a single form
  posting `active=true|false`. Deletion is intentionally not exposed —
  pausing preserves history.

### Step 5 — Patient view

- The patient route group is **fully unauthenticated**. The patient UUID
  in the URL is the capability. Drizzle uses a privileged DB connection
  that bypasses RLS for these reads (RLS is still in place as defense in
  depth for the caregiver routes, where the connection is tied to the
  signed-in user via Supabase auth).
- `lib/branding.ts` `getPatientBrand(patient?)` is the single source of
  truth for the patient-visible bank name. M1 returns the constant
  `"Clearview Savings"`. Per-patient brand customisation (Milestone 5+) replaces the
  body of this function only — every call site is already routed through
  it.
- **Unknown patient UUIDs render a calm welcome fallback**, never a 404.
  This is required by the patient UX rules in `CLAUDE.md`.
- The greeting (`Good morning|afternoon|evening`) reads `now.getHours()`
  in **server time**, not patient timezone. For a single-host dev setup
  this is fine; if patients across timezones become a thing, store
  timezone in `patients.settings` and use it here.

### Step 6 — Materialize tests

- Tests run against `pg-mem` (in-memory Postgres). Two compatibility
  shims live in `lib/test/pg-mem.ts`:
  1. pg-mem doesn't ship with `pgcrypto`, so we register a JS-backed
     `gen_random_uuid()` to satisfy the migration's PK defaults.
  2. pg-mem's parser rejects `ON CONFLICT (cols) WHERE pred DO NOTHING`,
     which Drizzle emits to match the partial unique index. The shim
     drops the predicate before passing to pg-mem; pg-mem treats unique
     indexes as non-partial, which gives the same effective semantics
     for this schema.
- These shims are scoped to test only — production uses real Postgres
  via `postgres.js`, which honours both pieces correctly.

### Step 7 — Seed

- `pnpm seed` is fully idempotent: re-running wipes the demo caregiver's
  patients (cascade kills accounts/transactions/scheduled-deposits/
  deposit-codes) and re-creates them.
- The seeded `Pension` is anchored to the **first of the current
  calendar month**, so on the next page load (caregiver or patient) the
  materializer fires once and posts the first $1,800 deposit. This is
  intentional — it demonstrates materialisation end-to-end on the very
  first interaction with the demo. The brief said "checking $1,200" as
  the seed _initial_ state; that is what is inserted, but expect the
  visible balance to be $3,000 after first page load.
- Demo credentials default to `demo@clearview-savings.test` / `DemoPassword123!`,
  overridable via `CLEARVIEW_SAVINGS_DEMO_EMAIL` / `CLEARVIEW_SAVINGS_DEMO_PASSWORD`.

## Conventions established in M1

These are decisions other milestones should follow without re-litigating.

- **Money is integer cents.** `bigint` columns, `mode: "number"` in
  Drizzle. Never floats. Currency formatting reads
  `patients.settings.currency` and `patients.settings.locale`.
- **Dates posted to scheduled deposits use `YYYY-MM-DD` strings**, not
  Date objects. The pg-mem test client coerces midnight-UTC Date columns
  back to strings to match `postgres.js` behaviour in production.
- **Server Actions, not REST endpoints**, for every mutation. Each action
  validates with a co-located Zod schema and uses `redirect(...)` to
  signal success/failure with `?status=...` or `?error=...` query
  params (we render those as inline messages — no toasts, ever).
- **Branding through `getPatientBrand()`.** No string literal `"Clearview Savings"` in
  patient-facing components.
- **Multi-tenant from day one.** Every caregiver query scopes by the
  current caregiver's id (via `getCurrentCaregiver()` and
  `getPatientForCaregiver()`); RLS enforces the same scoping at the
  database level. Don't trust a `patientId` from the client without
  routing through `getPatientForCaregiver()`.
- **`amountCents` is signed delta to balance.** See deviations § Step 4.

## TODOs deferred to later milestones

Things that the brief or `CLAUDE.md` mention but were intentionally not
built in M1.

- **Audit log table** for caregiver adjustments (date, before, after,
  note). The schema and dashboard have no audit infrastructure today.
  Likely M2 or M3 if/when caregivers need a paper trail.
- **Per-patient bank brand** (`patient.settings.brand_name`,
  `brand_logo_url`, etc.) — Milestone 5+. Helper is in place; only the
  body of `getPatientBrand` and a UI to set the value need to land.
- **Patient settings UI**: today, `patients.settings` is hardcoded by
  the seed and the `addPatientAction` (default `font_size: lg`,
  `locale: en-US`, `currency: USD`). No way to change it from the
  caregiver dashboard yet.
- **Patient timezone**: the greeting uses server time. Future: store
  timezone in `patients.settings` and use it for the greeting and for
  any date/time display.
- **`Deposit a Check` patient flow** and **single-use deposit codes**
  for printed checks/workbooks. Planned for M2.
- **Caregiver "view as patient" mode**: today, "Switch to patient view"
  opens `/patient/[id]` in a new tab. The brief mentions a confirmation
  step before switching; not implemented because the new-tab pattern
  already prevents accidental navigation.
- **Richer demo data** in the seed (a handful of historical
  transactions). Today the seed only inserts the $1,200 opening
  balance and the Pension schedule.
- **Email & confirmation flow polish**: the sign-up "check your email"
  state is functional but minimal. No resend-confirmation flow,
  password-reset flow, or email-change flow.
