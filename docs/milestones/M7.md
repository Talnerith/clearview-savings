# Milestone 7 — Caregiver account MFA (TOTP two-factor)

Built on top of M6 (auth hardening, commits ending at `25bf86f` on
2026-05-24). M6 layered CAPTCHA + per-email lockout over M5's per-IP
limiter, stopping the abuse classes that bypass a single-IP gate. M7
adds the second factor itself: an opt-in authenticator-app TOTP code so
a leaked or guessed password alone can no longer reach a caregiver's
patients, with one-time recovery codes for the lost-phone case.

The frozen pre-flight contract is `docs/specs/M7.md`; this file is the
implementation plan and records implementation-time decisions that
deviate from or extend that contract.

## Spec deviations / corrections recorded here

### No request-reachable privileged client exists yet (2026-05-28)

The spec Risks section and the Resolved item #3 say to "reuse the
existing privileged call-site pattern (see `docs/security/rls-audit.md`)."
On inspection, the **only** `SUPABASE_SECRET_KEY` call site in the repo
is `scripts/seed.ts:85` — a dev-only script never imported by a request
handler (confirmed in `docs/security/rls-audit.md` § "Service-role /
privileged-DB call sites": *"No request-handler use of the service
role"*). There is no `lib/supabase/admin.ts` to reuse.

So M7 **creates** a new server-only privileged module
(`lib/supabase/admin.ts`) rather than reusing an existing one. It copies
seed.ts's exact construction —
`createClient(url, SUPABASE_SECRET_KEY, { auth: { autoRefreshToken: false, persistSession: false } })`
— which is the "pattern" the spec meant. This is the first
request-reachable privileged call site in the app; it is `import
"server-only"` and used by exactly one code path (lost-device factor
removal). Recorded here because it sharpens the spec's "reuse" wording
into "copy the pattern into a new, isolated module."

### Password-reset criterion is satisfied by the middleware AAL check (2026-05-28)

The spec acceptance criterion *"a password reset does not bypass MFA"*
needs no change to `app/(auth)/reset-password/actions.ts`. That action
redirects to `/caregiver` after `updateUser` (line 42), and the
recovery-link session is **AAL1**. Once Step 5 adds the AAL2 gate to
middleware, an AAL1 session with a verified factor hitting `/caregiver`
is bounced to the challenge automatically — the reset flow inherits the
enforcement. The reset action stays as-is; the criterion is covered by a
**test** asserting the post-reset session lands at AAL1 and is
challenged, not by touching the action. Recorded so a future session
doesn't "fix" reset-password to redirect to the challenge directly.

#### Correction — that premise was wrong; reset needs a step-up (2026-06-06)

The Step 8 production smoke (Phase 7 / criterion F) disproved the note
above. The deviation assumed `updateUser({ password })` **succeeds** at
AAL1 and the `/caregiver` redirect then triggers the middleware
challenge. It does not: when the caregiver has MFA enabled, **Supabase
itself rejects `updateUser` at AAL1** with *"AAL2 session is required to
update email or password when MFA is enabled."* So the password is never
set, the recovery-link caregiver is dead-ended on the reset screen, and
the raw SDK string leaked into the UI.

Security-wise criterion F still holds — the reset genuinely cannot
bypass MFA — but the legitimate flow was broken. Resolved by a
**step-up**: `reset-password` (page loader + a defensive guard in the
action) detects `getAalState() === "aal1-needs-aal2"` and routes through
`/challenge?next=/reset-password`; a validated `next` param
(`lib/auth/next-path.ts` `safeNextPath`) lets `verifyChallengeAction`
return to the reset form once the session is AAL2. This *requires* the
authenticator on top of email access — strengthening, not bypassing,
MFA. The residual `updateUser` error is mapped to a calm message rather
than leaking the SDK string. So a future session **should** keep the
reset → challenge step-up; the original "don't redirect to the
challenge" guidance is superseded by this note.

## Resolved (from the spec, locked at freeze on 2026-05-24)

The spec's six open questions resolved at freeze (full text in
`docs/specs/M7.md` § Resolved):

1. **Enforcement — opt-in.** No forced enrollment; enabling MFA is a
   deliberate choice in settings. Org-wide-require toggle is M8+.
2. **Recovery — one-time codes + self-serve unenroll.** Codes at
   enrollment (salted-hash stored); using one unenrolls the lost factor
   via the privileged admin API, drops to AAL1, prompts re-enrollment.
   Admin runbook is the backstop when codes are also lost.
3. **Privileged key — existing `SUPABASE_SECRET_KEY`.** No new env var.
   (See deviation note above on the call-site pattern.)
4. **Failed-TOTP limiting — Supabase built-in.** No custom limiter or
   `mfa-lockout` notification in M7.
5. **Disable MFA — re-verify + notify.** Fresh TOTP/recovery code
   required to disable; fires a `mfa-disabled` admin notification.
6. **Security-page location — `/caregiver/settings`.** General settings
   page with a Security section.

### Step 2 — `mfa_recovery_codes` table + migration + RLS

Supabase TOTP has **no native recovery codes** (spec Risk #1), so we
build our own table. Drizzle schema additions to `lib/db/schema.ts`,
mirroring the existing table conventions (uuid PK `defaultRandom`,
`caregiver_id` FK with `onDelete: "cascade"`, `timestamptz`
`defaultNow`):

```ts
export const mfaRecoveryCodes = pgTable(
  "mfa_recovery_codes",
  {
    id: uuid("id").primaryKey().defaultRandom(),
    caregiverId: uuid("caregiver_id")
      .notNull()
      .references(() => caregivers.id, { onDelete: "cascade" }),
    // scrypt-derived, per-code random salt embedded:
    // "scrypt$<saltHex>$<hashHex>". Never stores the plaintext code.
    codeHash: text("code_hash").notNull(),
    usedAt: timestamp("used_at", { withTimezone: true }),
    createdAt: timestamp("created_at", { withTimezone: true })
      .notNull()
      .defaultNow(),
  },
  (t) => [index("mfa_recovery_codes_caregiver_idx").on(t.caregiverId)],
);
```

Plus `MfaRecoveryCode` / `NewMfaRecoveryCode` inferred types at the
bottom of the file alongside the existing exports.

**Migration** — next sequential file is `drizzle/0004_*.sql` (current
head is `0003_first_freak.sql`). Generate via `pnpm db:push` /
drizzle-kit so the `drizzle/meta` snapshot (`0004_snapshot.json` +
`_journal.json`) stays consistent. Do not hand-author the SQL.

**RLS policy** — add to `supabase/policies.sql`, same shape as
`audit_log_owner` (scoped by `caregiver_id` through the caregivers
ownership chain):

```sql
alter table public.mfa_recovery_codes enable row level security;
create policy mfa_recovery_codes_owner on public.mfa_recovery_codes
  for all to authenticated
  using (caregiver_id in (select id from public.caregivers where user_id = auth.uid()))
  with check (caregiver_id in (select id from public.caregivers where user_id = auth.uid()));
```

As with every other table, RLS is dormant in production (the app reads
through the `postgres`-superuser `db` connection — see rls-audit.md
Finding 2); the policy is defense-in-depth for the day a query reaches
for the Supabase JS data path. The application layer (Step 3 lib, scoped
by `caregiver_id`) is the live gate.

No pause — Step 3 depends directly on this table.

### Step 3 — Privileged admin client + recovery-code lib + tests

**`lib/supabase/admin.ts`** (new, `import "server-only"`). Copies the
seed.ts construction. Single export:

```ts
export function createSupabaseAdminClient(): SupabaseClient {
  return createClient(
    requireEnv("NEXT_PUBLIC_SUPABASE_URL"),
    requireEnv("SUPABASE_SECRET_KEY"),
    { auth: { autoRefreshToken: false, persistSession: false } },
  );
}
```

Used by exactly one path (Step 6 lost-device unenroll →
`auth.admin.mfa.deleteFactor({ id, userId })`). Never imported by a
client component; `server-only` enforces that at build time.

**`lib/mfa/recovery-codes.ts`** (new, `server-only`). The recovery-code
domain logic, scoped by `caregiverId` (never trusts a client value —
the caller passes the caregiver resolved from the session):

- `generateRecoveryCodes(caregiverId): Promise<string[]>` — generates
  **10** codes (format: 10 chars, Crockford base32, grouped
  `xxxxx-xxxxx` for legibility), scrypt-hashes each with a per-code
  random 16-byte salt, **deletes any existing rows** for the caregiver,
  inserts the new hashed set, and returns the **plaintext** codes for
  one-time display. Plaintext never persists.
- `consumeRecoveryCode(caregiverId, submitted): Promise<boolean>` —
  normalizes the submitted code (strip whitespace/dashes, uppercase),
  fetches the caregiver's **unused** rows, scrypt-compares each via
  `crypto.timingSafeEqual`, and on a match sets `used_at` (single-use)
  and returns `true`. No match → `false`. N ≤ 10 so the linear scan is
  trivial; we can't index by hash because each row has its own salt.
- `countUnusedRecoveryCodes(caregiverId): Promise<number>` — for the
  settings page to show "N codes remaining."

Hashing helper (private): `scrypt` from `node:crypto` (no new
dependency), `scryptSync(code, salt, 32)`, stored as
`scrypt$<saltHex>$<hashHex>`. Codes are high-entropy (50 bits), so
scrypt's work factor is belt-and-suspenders, not load-bearing — but it
matches the spec's "salted hashes" and costs nothing.

**Regeneration invalidates the old set** by construction:
`generateRecoveryCodes` deletes-then-inserts, so the prior hashes are
gone the moment a new set is issued (spec acceptance criterion).

**Tests** — `lib/mfa/recovery-codes.test.ts` (mocks the `db` layer):
- `generateRecoveryCodes` returns 10 plaintext codes and stores 10
  hashed rows; no plaintext appears in any stored value.
- a freshly generated code `consumeRecoveryCode`s to `true` once, then
  `false` on reuse (single-use; `used_at` set).
- a wrong/garbage code returns `false` and consumes nothing.
- `generateRecoveryCodes` a second time deletes the old rows (old codes
  stop verifying).
- code normalization — `ABCDE-FGHIJ`, `abcde fghij`, `abcdefghij`
  collapse to the same match.

No pause — Step 4 consumes this lib.

### Step 4 — `/caregiver/settings` Security section + enrollment

The caregiver settings surface does **not** exist today
(`app/(caregiver)/layout.tsx` is dashboard + sign-out only). M7 creates
it.

**New route** `app/(caregiver)/caregiver/settings/page.tsx` — a settings
page with a **Security** section. Standard caregiver density (not
patient UX). Add a "Settings" link to the caregiver header in
`app/(caregiver)/layout.tsx` beside the existing Brandmark / Sign-out.

**Enrollment flow** (Supabase MFA: `enroll` → `challenge` → `verify`):

1. Security section shows current MFA state via
   `supabase.auth.mfa.listFactors()` — "Two-factor authentication is
   off" with an **Enable** action, or "On" with **Regenerate recovery
   codes** / **Disable** actions (Disable is Step 6).
2. **Enable** → server action `enrollTotpAction`:
   `supabase.auth.mfa.enroll({ factorType: 'totp', friendlyName })`
   returns `{ id, totp: { qr_code (SVG data URI), secret, uri } }`.
   Render the QR `<img>` + the copyable manual `secret`, and a 6-digit
   code input. Store the returned `factorId` in the form (hidden field)
   for the verify step.
3. **Verify** → server action `verifyEnrollmentAction(factorId, code)`:
   `challenge({ factorId })` → `verify({ factorId, challengeId, code })`.
   On success the factor becomes verified **and** the session steps up
   to AAL2. Then call `generateRecoveryCodes(caregiver.id)` and render
   the 10 codes **once** with a clear "save these now — they will not be
   shown again" prompt (printable; inline on the page, not a modal —
   though this is caregiver UX, we keep it inline for consistency).
4. Wrong code → calm inline error, factor stays unverified, no recovery
   codes generated. Repeated wrong codes hit Supabase's built-in
   MFA-verify rate limit (spec Resolved #4 — no custom limiter).

**Regenerate recovery codes** → server action `regenerateCodesAction`:
requires the session already be AAL2 (the caregiver is in settings, which
Step 5 gates at AAL2), calls `generateRecoveryCodes` (deletes old,
issues new), shows the new set once.

Zod schemas live next to the action (6-digit numeric code; `factorId`
uuid). Codes are shown via a server-rendered one-time view — never
re-fetchable.

**Pause** after Step 4 so the user can manually enroll a real
authenticator app against the live Supabase project and confirm the QR /
secret / verify / recovery-code-display loop works before we make MFA
*enforced* at sign-in (Step 5). At this point a verified factor exists
but sign-in does not yet challenge — deliberate, so a bug in enrollment
can't lock the user out before the challenge path is tested.

### Step 5 — Sign-in two-step + `/challenge` page + AAL2 enforcement

This is the load-bearing security step: it makes a verified factor
actually gate the dashboard. Two halves land together because the
challenge page is useless without enforcement and enforcement without
the challenge page locks out any caregiver with a factor.

**AAL helper** `lib/auth/aal.ts` (new, `server-only`) — a thin wrapper
so middleware, loaders, and pages share one source of truth:

```ts
// "aal1-needs-aal2" = verified factor exists but session is AAL1.
type AalState = "no-factor" | "aal1-needs-aal2" | "aal2";
export async function getAalState(supabase): Promise<AalState>;
```

Built on `supabase.auth.mfa.getAuthenticatorAssuranceLevel()`:
`currentLevel === 'aal2'` → `"aal2"`;
`currentLevel === 'aal1' && nextLevel === 'aal2'` → `"aal1-needs-aal2"`;
otherwise → `"no-factor"`. This is a local computation off the session
(no network round-trip), cheap to call in middleware.

**Sign-in redirect** — `app/(auth)/sign-in/actions.ts` line 67. Today it
unconditionally `redirect("/caregiver")`. After a successful
`signInWithPassword`, branch on `getAalState`:
- `"aal1-needs-aal2"` → `redirect("/challenge")`
- otherwise → `redirect("/caregiver")` (unchanged for the no-factor
  caregiver — spec acceptance criterion #1: their flow is identical).

**Challenge page** `app/(auth)/challenge/page.tsx` (route `/challenge`,
in the `(auth)` group so it is **not** under `/caregiver` and therefore
not caught by the AAL2 redirect — avoids a redirect loop). Page loader:
`getUser()`; no user → `/sign-in`; `getAalState` is `"aal2"` →
`/caregiver` (already stepped up); `"no-factor"` → `/caregiver` (nothing
to challenge). Otherwise render:
- a 6-digit TOTP code input → `verifyChallengeAction`:
  `listFactors()` → first verified TOTP factor → `challenge` → `verify`.
  Success steps the session to AAL2 → `redirect("/caregiver")`. Wrong
  code → calm inline error (Supabase's built-in limit covers brute
  force).
- a "Use a recovery code instead" link → the recovery path (Step 6).

The existing middleware redirect-away-from-auth-pages check
(`path === "/sign-in" || path === "/sign-up"`, exact match) does **not**
catch `/challenge`, so a signed-in AAL1 user can reach it. Confirmed
against `lib/supabase/middleware.ts:54`.

**Middleware AAL2 enforcement** — `lib/supabase/middleware.ts`, beside
the existing `!user && path.startsWith("/caregiver")` redirect
(lines 46–51). After the `!user` check, when `user` is present and
`path.startsWith("/caregiver")`, compute `getAalState`; if
`"aal1-needs-aal2"`, redirect to `/challenge`. `"no-factor"` and `"aal2"`
pass through. Patient routes (`/patient/[id]`) are untouched — they are
not in any `/caregiver` branch and the matcher already excludes static
assets; reaffirmed, not modified (spec hard constraint).

**Server-layer enforcement** — `lib/auth/current-caregiver.ts`
`getCurrentCaregiver()` gains an AAL2 assertion after the `!user`
redirect: if `getAalState() === "aal1-needs-aal2"`, `redirect("/challenge")`.
This closes the spec Risk "AAL2 enforced at route layer only" — a stolen
AAL1 cookie that somehow bypasses middleware still can't load caregiver
data through the server loader. (RLS-level `aal2` gating stays an M8
item per the spec non-goal.)

**Tests:**

- middleware AAL2 redirect (mock the SSR client + `getAalState`):

### Step 6 — Lost-device recovery + disable MFA + notification

**Lost-device recovery** — on `/challenge`, the "Use a recovery code"
path → `recoverWithCodeAction(code)`:
1. `getUser()` (AAL1 session from the password step) → caregiver row.
2. `consumeRecoveryCode(caregiver.id, code)` — `false` → calm inline
   error, no state change.
3. On `true`: load the caregiver's verified TOTP factor
   (`listFactors()`), then **`createSupabaseAdminClient()` →
   `auth.admin.mfa.deleteFactor({ id: factorId, userId })`** to unenroll
   the lost factor with the privileged key (the normal `unenroll`
   requires AAL2, which a locked-out caregiver doesn't have — this is
   why the privileged path exists).
4. The session is now AAL1 with **no** factor → `getAalState` becomes
   `"no-factor"` → the caregiver passes the middleware gate.
   `redirect("/caregiver?reenroll=1")`; the dashboard shows a calm inline
   banner prompting re-enrollment in settings.

A recovery code cannot itself mint AAL2 (Supabase only grants AAL2 from
a TOTP verify); this unenroll-then-re-enroll path is the honest
mechanism, recorded in the spec.

**Disable MFA** — from `/caregiver/settings`, `disableMfaAction`. The
caregiver is already AAL2 to be in settings (Step 5 gate). Requires a
**fresh** re-verification before unenrolling (spec Resolved #5 — turning
off protection is a security event):
- a fresh TOTP code (`challenge` → `verify`), **or** a valid recovery
  code (`consumeRecoveryCode`).
- on success: `supabase.auth.mfa.unenroll({ factorId })` (normal API —
  AAL2 present), delete the caregiver's `mfa_recovery_codes` rows (moot
  once the factor is gone), and fire
  `sendAdminNotification({ kind: "mfa-disabled", caregiverEmail, caregiverId })`
  wrapped in try/catch + Sentry (same shape as the `new-caregiver` send
  in `current-caregiver.ts:46-54`).

**`mfa-disabled` notification kind** — extend the M6 union exactly as M6
extended M5's:
- `emails/admin-notification.tsx`: add
  `| { kind: "mfa-disabled"; caregiverEmail: string; caregiverId: string }`
  to `AdminNotificationProps`, plus cases in `previewFor`, `titleFor`,
  `bodyFor`. Body: states the caregiver disabled two-factor and returned
  to single-factor sign-in; lists email + caregiver ID; notes that if
  unexpected it may indicate account compromise.
- `lib/admin-email.ts`: add a `subjectFor` case —
  `` `MFA disabled — ${n.caregiverEmail}` ``.
- `emails/admin-notification.test.tsx`: add a snapshot for the
  `mfa-disabled` kind (the existing five snapshots stay untouched).

Type-check enforces all six kinds are handled in every switch.

**Tests:**
- `recoverWithCodeAction` — valid code unenrolls via the (mocked) admin
  client, consumes the code, redirects with the re-enroll flag; invalid
  code → inline error, no admin call.
- `disableMfaAction` — fresh TOTP path and recovery-code path each
  unenroll + fire `mfa-disabled`; without re-verification it refuses.
- `mfa-disabled` admin-notification snapshot.

## Re-anchor protocol (mid-milestone)

When starting a fresh session mid-M7:

1. Read `CLAUDE.md`
2. Read `docs/specs/M7.md` (the frozen pre-flight contract)
3. Read this file — `docs/milestones/M7.md`
4. Read `docs/milestones/M7-progress.md` (current state, written at the
   previous session's close)
5. Inspect actual files to confirm the progress doc matches reality
6. Run `pnpm typecheck && pnpm lint && pnpm test --run` for a green
   baseline
7. Wait for "go" before writing code

`scripts/re-anchor.sh 7` prints the above plus the last 10 commits and
current git status.
