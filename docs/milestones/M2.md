# Milestone 2 — Printable checks + single-use deposit codes

Built on top of `m1-complete` (commit `f08d3d2`), shipped in a single
`feat:` commit.

The goal: caregiver generates a printable check carrying a single-use
code; patient "deposits" it through the familiar three-step bank ritual
(photo → amount → code); the code redeems atomically, creating exactly
one transaction even under a race.

`pnpm typecheck && pnpm lint` clean. `pnpm test --run` — **12 passing**
(4 M1 + 8 M2).

## Decisions resolved at the start of the milestone

These two open questions were answered before any code was written, and
they shape several of the deviations below.

### Typed amount vs. code amount — _theatre_

**The patient-typed amount is theatre. The code's `amount_cents` is
authoritative.** People with dementia misread digits; rejecting a
patient who reads `$100.00` as `$1.00` produces exactly the distress
Clearview Savings exists to prevent. The caregiver already decided the amount when
they generated the check, and that value is printed on the check in
numerals _and_ words. The "type the deposit amount" step is preserved
because it is a familiar bank ritual that gives the patient agency —
its output is ignored.

Implementation: `redeemCode({ patientId, code })` structurally excludes
a typed amount; `redeemCodeAction` never reads `formData.get("amount")`.
A row-level test pins the contract: when the typed amount differs from
the code's, the resulting transaction matches the code.

### Per-patient daily/weekly deposit limits — _deferred_

**Not built in M2.** The risk this guard mitigates ("patient finds a
stack of old printed checks and tries to deposit them all") only emerges
after weeks of real use, and we have no production caregivers yet.
`CLAUDE.md` is explicit on not designing for hypothetical futures.
`redeemCode` is the single call site and `patients.settings` is jsonb
— adding it later is a small change. Listed as a TODO below.

## Deviations from the original plan

### Step 1 — atomicity without `db.transaction(...)`

The plan wrapped the multi-step claim/insert/update sequence in
`db.transaction(...)`. **The shipping code does not.** `pg-proxy` (used
by the test harness) throws unconditionally for transactions —
`pg-proxy/session.js`: "Transactions are not supported by the Postgres
Proxy driver". Atomicity comes instead from the atomic
`UPDATE deposit_codes SET status='used' WHERE status='unused' RETURNING`
— only one caller can claim. The remaining writes (insert tx, update
balance, backfill `transaction_id`) sequence after. There is a small
consistency window if the process dies mid-sequence; a reconciliation
hint is documented in a comment by the function. Pg-mem serialises
queries (single-threaded), so the concurrency test gives a true result.

A single-CTE alternative (claim + insert in one `WITH ... INSERT`) was
attempted and abandoned — pg-mem rejects comma cross-joins, `CROSS
JOIN`, and `INSERT` inside CTE-with-scalar-subquery (internal pg-mem
bug at `insert.ts:59`). Switching the test backend to
`drizzle-orm/node-postgres + pg-mem` was also tried and abandoned —
pg-mem rejects `types.getTypeParser` (always sent by drizzle) and
`rowMode: "array"`.

### Step 1 — migration tracking

The `deposit_codes.label` NOT NULL column was added with the safe
default-then-drop pattern (`ADD COLUMN ... DEFAULT '' NOT NULL`, then
`ALTER COLUMN ... DROP DEFAULT`) so the constraint can be added on a
table that may already have rows. Drizzle generated a single ALTER; the
migration was hand-edited to two statements.

`scripts/apply-migration.ts` now tracks applied migrations in a
`_b4a_applied_migrations` table rather than relying on Drizzle's own
journal — the journal got out of sync when `drizzle-kit push` failed on
Windows in M1, so we own the tracking explicitly now. The existing
`0000_*` was backfilled into the table by an inline node script using
`postgres.js`.

### Step 2 — PDF stream type plumbing

`@react-pdf/renderer`'s `renderToStream` returns
`Promise<NodeJS.ReadableStream>` (not `Readable`). The route bridges
Node→Web stream via `Readable.toWeb(nodeStream as Readable)` (Node 18+
native). Returned with `Content-Disposition: inline` so the browser's
print dialog opens directly in the tab.

### Step 3 — page split for metadata

The plan said `/patient/[id]/deposit/page.tsx` is a client component.
Next.js metadata only works in RSCs and we need a custom tab title, so
the page was split: `page.tsx` (server: patient lookup, brand header,
`generateMetadata`) mounts `<DepositWizard patientId={...} />` (client:
3-step `useState` wizard). `useFormStatus` (not `useTransition`)
handles the pending state — cleaner match for `<form
action={serverAction}>`.

### Step 5 — `createCheckAction` redirect target

The plan asked the action to `redirect(... /pdf)` _and_ to show a
`?status=...` banner on the list page "after returning from the PDF".
Those are in tension: a server-action redirect can't usefully land on a
binary PDF stream and then return to a banner-bearing URL.

**Resolution:** action redirects to the list with
`?status=just-generated&codeId=<id>`. The banner there carries an "Open
the printable check ↗" button (`target="_blank"`) that opens the PDF in
a new tab while keeping the list available. Same outcome (PDF opens),
one extra click, more reliable UX, no special infrastructure beyond a
query banner. The list-page banner also resolves `justGenerated =
checks.find(c => c.id === codeId)` so a stale or hand-crafted URL
produces no banner.

### Step 6 — branding pass swept up the M1 home page

Initial grep found three `"Clearview Savings"` literals in `app/(patient)/**` —
all in static `metadata` exports, including the M1 home page. M2's
strict scope was the M2 routes, but the same pass also asked for a
clean grep, so the M1 home page was brought in line. `lib/branding.ts`
gained `getPatientBrandById(id)` (UUID-validates, looks up the patient,
falls back to default brand on unknown id). All three patient pages now
use `async generateMetadata({ params })`. Only remaining `"Clearview Savings"`
literal reachable from a patient render path is `DEFAULT_BRAND` in
`lib/branding.ts` — exactly what the rule requires. Forbidden patient
terms ("Alzheimer", "dementia", "simulat", "therapeutic", `\bfake\b`,
`\bdemo\b`) — 0 matches.

## Conventions established in M2

- **`redeemCode`'s signature is `{ db, patientId, code }`** —
  structurally excludes a typed amount. Don't add an `amount` parameter
  here; the code is the source of truth.
- **Single error reason for every redemption failure path.** Bad UUID,
  malformed code, code-not-found, code-already-used, and `redeemCode`
  throws all redirect to the same `?error=invalid_or_used`. Small but
  defensible privacy choice — don't add per-reason messages.
- **Server actions re-validate even unauthenticated patient routes.**
  The wizard hidden-field UUID is re-checked against the patient row.
  Defense in depth.
- **Done-page lookups join on `accounts.patient_id`.** A hand-crafted
  `?txId=<otherPatientTx>` would otherwise leak someone else's
  transaction. Apply the same pattern when adding any future patient
  detail page that takes an id from the URL.
- **`label` and `memo` are separate columns on `deposit_codes`.**
  `label` is the caregiver's name for the check that becomes the
  transaction's display label on Recent Transactions. `memo` is the
  printed-on-check "for ___" line — separate concern, nullable.
- **No emoji decorations in patient UI.** Per CLAUDE.md "Only use
  emojis if the user explicitly requests it" — initial draft had a 📷
  in the photo step and a ✓ on the done page; both were removed.

## TODOs deferred to later milestones

- **Per-patient daily/weekly deposit limits.** Single call-site seam in
  `redeemCode`; jsonb on `patients.settings` means no migration. See
  rationale in "Decisions resolved" above.
- **Voiding a code.** `deposit_codes.status` enum is currently
  `unused | used`; adding `voided` is a one-value migration. The
  redemption query already filters by `status = 'unused'`.
- **Multi-account routing.** `redeemCode` deposits to the first account
  by `created_at asc`. Adding a `target_account_id` on the code (or a
  setting on the patient) is additive.
- **Real check images / OCR.** The patient camera step deliberately
  reads to memory and discards. If a future feature needs the image,
  this is the right place to introduce upload — but only when there's
  a concrete reason.
- **Audit log table** (still — carried over from M1).
- **Workbooks** — Milestone 3.
