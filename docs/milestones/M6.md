# Milestone 6 — Auth hardening

Built on top of M5 (production launch, commits ending at `bf25889`
on 2026-05-19). M5 left a per-IP sliding-window rate limiter on
sign-up / sign-in / forgot-password and an admin-notification
pipeline; M6 layers CAPTCHA and per-email lockout on top so the
attack classes that bypass per-IP limiting — distributed
credential stuffing, automated sign-up abuse — are also stopped.

The frozen pre-flight contract is `docs/specs/M6.md`; this file
is the implementation plan and records implementation-time
decisions that deviate from or extend that contract.

### Step 7 smoke deviation — double Turnstile verification (2026-05-24)

The Step 7 production smoke surfaced a bug: a solved widget still
failed account creation with "request disallowed." Root cause —
the single-use Turnstile token was redeemed twice: once by
`lib/turnstile.ts`'s `siteverify` call (Step 2) and again by
Supabase Auth via `captchaToken` (Step 5). The second redemption
fails `timeout-or-duplicate`, so GoTrue rejected every real
submission. Invisible to tests/dev because the verifier bypassed
when `NODE_ENV !== "production"` and the integration test mocked
it; only the live Cloudflare API exercises both redemptions.

Resolution (user-locked): **Supabase Auth is the sole verifier.**
`lib/turnstile.ts` no longer calls `siteverify`; it now exposes
`checkTurnstilePresent` (presence-only short-circuit) and
`isCaptchaRejection` (maps GoTrue's "request disallowed" to the
calm message). The three actions keep forwarding `captchaToken`
and the Supabase dashboard toggle stays on. Full rationale and the
rejected alternatives are in **ADR 0003**. Note: this retires the
planned `TURNSTILE_TEST_BYPASS_KEY` E2E path (a sentinel token
can't pass GoTrue) — deferred follow-up, does not block the smoke.

## Resolved (from the spec, locked at freeze on 2026-05-19)

The spec's six open questions resolved at freeze:

1. **CAPTCHA provider** — Cloudflare Turnstile (free, privacy-
   first, Supabase-native).
2. **Per-email lockout** — 5 failed `/sign-in` attempts per email
   per 15-minute sliding window → 15-minute lockout.
3. **Vercel Pro / WAF** — skip Pro, stay on Hobby. WAF moves to
   non-goals.
4. **Lockout notification** — silent to the locked-out account
   holder, admin email only, one per lockout event.
5. **CAPTCHA on `/reset-password`** — no. Signed reset URL is the
   security boundary; widget would friction recovery without
   meaningful security gain.
6. **E2E CAPTCHA bypass** — `TURNSTILE_TEST_BYPASS_KEY` env var
   gated to a known sentinel token; set in Vercel Preview only,
   never Production.

### Step 1 — Turnstile account + Supabase Auth dashboard toggles

User-driven setup, paired with the Step 1 pause prompt. Three
artifacts produced:

1. **Cloudflare Turnstile site** — user visits
   https://dash.cloudflare.com → Turnstile → Add site. Domain
   `clearviewsavings.com` (plus localhost in dev). Widget mode

   copies the **site key** (public) and **secret key** (server).
2. **Supabase Auth CAPTCHA toggle** — user visits Supabase
   dashboard → Authentication → Settings → Bot and Abuse
   Protection. Enable CAPTCHA, choose Turnstile, paste the
   secret key. Save.
3. **Env vars added** to Vercel project (Production env) and
   `.env.local` (dev):
   - `NEXT_PUBLIC_TURNSTILE_SITE_KEY` — site key (client-visible
     by design; the `NEXT_PUBLIC_` prefix is required for the
     React widget mount)
   - `TURNSTILE_SECRET_KEY` — secret key (server-only)
   - `TURNSTILE_TEST_BYPASS_KEY` — random 32-char hex, generated
     by user. Set **only in Vercel Preview** env, **never
     Production**. Optional in dev (only matters for the Preview
     E2E path).

`.env.example` updated with the three keys + comments explaining
which environments each lives in.

**Why `Managed` mode** and not `Invisible` or `Non-interactive`:
Managed is the accessibility-friendly default — most users see no
interaction (invisible challenge passes silently), but the widget
falls back to an audible / visual challenge when Cloudflare's
risk signals demand it. Invisible mode would be friendlier still
but doesn't gate well against persistent automated abuse, which
is exactly what M6 targets.

### Step 2 — `lib/turnstile.ts` server-side verifier

New file. Single exported function:

```ts
export async function verifyTurnstileToken(
  token: string | null | undefined,
): Promise<{ ok: true } | { ok: false; reason: string }>;
```

Behavior:

- **Non-production bypass:** if `process.env.NODE_ENV !==
  "production"`, return `{ ok: true }` without calling
  Cloudflare. Same gate as the existing per-IP limiter — dev and
  CI never block on Turnstile. Documented and intentional.
- **Preview-env test bypass:** if `process.env
  .TURNSTILE_TEST_BYPASS_KEY` is set and `token` exactly equals
  that string, return `{ ok: true }` without calling Cloudflare.
  The env var is set only in Vercel Preview deploys (per Step
  1), so Production cannot be bypassed even with knowledge of
  the sentinel.
- **Missing-secret fall-safe:** if `TURNSTILE_SECRET_KEY` is
  unset in production, return `{ ok: false, reason:
  "captcha_misconfigured" }` and log the misconfig via Sentry.
  This is fail-**closed**: a misconfigured production cannot
  silently accept un-CAPTCHA-verified submissions. The risk is
  documented in the spec (Turnstile / Cloudflare outage stops
  sign-up entirely; bounded calm-error UX) and matches the
  spec's closed-fail framing.
- **Otherwise:** POST to
  `https://challenges.cloudflare.com/turnstile/v0/siteverify`
  with `secret` + `response` (the token) + `remoteip` (the
  client IP from the existing `getClientIdentifier()`). Parse
  the JSON. On `success: true` return `{ ok: true }`; on
  `success: false` return `{ ok: false, reason: <first
  error-code> }`.
- **Network failure / 5xx:** catch the fetch error, return `{
  ok: false, reason: "captcha_unavailable" }`, log via Sentry.
  Same closed-fail rationale.

**Test file** `lib/turnstile.test.ts`:

- non-production bypass returns `{ ok: true }` without fetch
- preview test-bypass sentinel returns `{ ok: true }` without
  fetch
- production with missing secret returns `{ ok: false, reason:
  "captcha_misconfigured" }` and captures to Sentry
- production with valid token (mocked fetch returning `success:
  true`) returns `{ ok: true }`
- production with invalid token (mocked `success: false, error-
  codes: ["invalid-input-response"]`) returns `{ ok: false,
  reason: "invalid-input-response" }`
- production with network failure (mocked fetch throwing)
  returns `{ ok: false, reason: "captcha_unavailable" }` and
  captures to Sentry

`__resetTurnstileCacheForTests()` follows the same pattern as
`__resetRateLimitCacheForTests()` for env-driven module caches.
If the module has no cached state (likely — the verifier reads
env per call), the reset helper is omitted.

### Step 3 — Per-email lockout in `lib/rate-limit.ts`

Two new exported functions, one new internal helper:

```ts
export async function checkEmailLockout(
  email: string,
): Promise<{ allowed: boolean }>;

export async function recordFailedSignIn(
  email: string,
): Promise<void>;
```

Storage model (Upstash Redis keys, parallel to the existing
`rl:breach:*` pattern):

- `rl:email-fails:<lc-email>` — `INCR`-counted, 15-min TTL
  sliding window via TTL reset on each increment. On reaching
  `EMAIL_LOCKOUT_THRESHOLD` (5), the helper sets the lockout key
  and fires the admin notification.
- `rl:email-lock:<lc-email>` — presence-only key, TTL 15 min.
  Existence = "this email is locked." `checkEmailLockout`
  returns `{ allowed: false }` when this key exists.

**Email normalization:** lowercased + trimmed before hashing
into the key. Different IPs hitting the same email get
correlated; different email casings of the same address get
correlated. We don't apply Gmail-style dot/plus folding —
Supabase Auth treats `a@x` and `a+b@x` as distinct addresses, so
we mirror that.

**Atomic increment + threshold** — same `INCR` pattern as M5
breach tracking. The lockout-set happens on the increment that
returns exactly `EMAIL_LOCKOUT_THRESHOLD` (5). Concurrent
requests cannot double-fire the lockout-set because only one of
them sees `count === 5`. Same atomicity guarantee covers the
admin-notification dedupe — one admin email per lockout event,
not one per blocked attempt during the lockout window.

**Wiring into `signInAction`:**

```ts
// inside signInAction, after parsing the form, before
// supabase.auth.signInWithPassword:

const { allowed: emailAllowed } = await checkEmailLockout(parsed.data.email);
if (!emailAllowed) {
  redirect("/sign-in?error=rate_limited");
}

// ... existing IP limiter + supabase call ...

if (error) {
  await recordFailedSignIn(parsed.data.email);
  redirect(`/sign-in?error=${encodeURIComponent(error.message)}`);
}
```

The per-email check runs **after** form parse (so we have the
email) and **before** the Supabase call. The per-IP check runs
first (Step 5 reorders these to keep them adjacent). On a
Supabase auth error, increment the fail counter. On success,
**do not increment** — a successful sign-in doesn't reset the
counter either (the TTL handles that naturally over 15 min).

**Non-production bypass** — same `NODE_ENV !== "production"`
gate as `checkRateLimit`. Local dev and CI never lock anyone
out. Returns `{ allowed: true }` immediately.

**Admin-notification path** — when the increment returns
exactly 5, call `sendAdminNotification({ kind: "email-lockout",
email, ip, attemptsInWindow: 5 })`. Wrapped in try/catch + Sentry
exactly like the existing breach path so the auth flow keeps
working if Resend or the admin mailbox is down. `ip` is captured
from `getClientIdentifier()` at recordFailedSignIn time — the
admin email gets the IP of the attempt that tripped the lockout,
which is the most useful single IP for triage.

**Tests added to `lib/rate-limit.test.ts`** (the existing per-IP
suite stays untouched):

- `checkEmailLockout` bypasses in non-production
- `checkEmailLockout` returns allowed when the lock key doesn't
  exist (mocked Redis `get` returning null)
- `checkEmailLockout` returns denied when the lock key exists
- `recordFailedSignIn` bypasses in non-production
- `recordFailedSignIn` increments the fail counter and sets TTL
  on the first failure
- `recordFailedSignIn` sets the lockout key + fires admin
  notification on the 5th failure
- `recordFailedSignIn` does **not** fire admin notification on
  failures 1–4 or 6+ within the window
- `recordFailedSignIn` swallows admin-notification errors to
  Sentry (same shape as the existing breach test)
- Email normalization — `Foo@Bar.com ` and `foo@bar.com`
  collapse to the same key

### Step 5 — Wire Turnstile widget + verifier into the three auth pages

**Client widget component** —
`components/auth/TurnstileWidget.tsx`. Client component.

```tsx
"use client";

import { Turnstile } from "@marsidev/react-turnstile";

export function TurnstileWidget() {
  const siteKey = process.env.NEXT_PUBLIC_TURNSTILE_SITE_KEY;
  if (!siteKey) {
    // dev / preview without a configured site key: the server
    // verifier bypasses in non-production, so the form still
    // submits. We render nothing rather than a broken widget.
    return null;
  }
  return (
    <Turnstile
      siteKey={siteKey}
      options={{ theme: "light", size: "normal" }}
    />
  );
}
```

The component reads the site key from the public env var. When
the widget mounts, it injects a hidden `cf-turnstile-response`
input into the surrounding `<form>` — that input is what the
server action picks up from `formData.get("cf-turnstile-
response")`.

**Install** `pnpm add @marsidev/react-turnstile` — wrapper over
the Cloudflare Turnstile JS that handles React lifecycle, ref
cleanup, automatic re-render on errors. The alternative is a
hand-rolled `<div className="cf-turnstile" data-sitekey={...}>`
+ a script tag; the wrapper saves ~30 lines of boilerplate and
handles the SSR-hydration edge case correctly.

**Page wiring** — three small page edits:

- `app/(auth)/sign-up/page.tsx` — add `<TurnstileWidget />`
  inside the `<form>`, after the password input, before the
  submit button.
- `app/(auth)/sign-in/page.tsx` — same placement.
- `app/(auth)/forgot-password/page.tsx` — same placement.

**Server-action wiring** — three edits to forward the token to
`lib/turnstile.ts` and then to Supabase Auth:

`app/(auth)/sign-up/actions.ts`:
```ts
const turnstileToken = formData.get("cf-turnstile-response")?.toString() ?? null;
const turnstile = await verifyTurnstileToken(turnstileToken);
if (!turnstile.ok) {
  redirect("/sign-up?error=Please+verify+and+try+again.");
}

// ... existing parse + IP-limiter checks ...

const { error } = await supabase.auth.signUp({
  ...parsed.data,
  options: {
    emailRedirectTo: `${origin}/caregiver`,
    captchaToken: turnstileToken ?? undefined,
  },
});
```

`app/(auth)/sign-in/actions.ts` — same pattern. `signIn
WithPassword({ email, password, options: { captchaToken } })`.

`app/(auth)/forgot-password/actions.ts` — same pattern.
`resetPasswordForEmail(email, { redirectTo, captchaToken })`.

Notes:

- The Turnstile verify call goes **before** the per-IP limiter
  check so an attacker can't burn the per-IP quota by
  submitting empty forms. Wait — actually the per-IP limiter
  itself catches that case (it counts attempts regardless of
  outcome). Sequencing locked at: Turnstile verify → per-IP
  limiter → per-email lockout (sign-in only) → form parse →
  Supabase call. Failure at any earlier step short-circuits
  later steps.
- The "please verify and try again" message uses the spec's
  exact phrasing. Render it on each auth page alongside the
  existing `?error=...` handler.
- `cf-turnstile-response` is the widget's fixed input name; not
  configurable.

**E2E preview-deploy path** — the Vercel Preview environment
has `TURNSTILE_TEST_BYPASS_KEY` set. The Playwright test (M5's
sign-up smoke flow) submits a hidden input value matching that
key as `cf-turnstile-response`, bypassing the real Turnstile
challenge. Production builds don't have the env var, so the
sentinel is meaningless there.

**Integration test** —
`app/(auth)/sign-up/actions.test.ts` (new file). Mocks
`lib/turnstile` + the Supabase client. Asserts:

- A missing / failed Turnstile token short-circuits with the
  calm error redirect; `supabase.auth.signUp` is **not**
  called.
- A passing Turnstile token forwards `captchaToken` to
  `supabase.auth.signUp` exactly as received.

One integration test for sign-up is enough — sign-in and
forgot-password follow the same template and the changes are
mechanical. The full Turnstile integration matrix lives in the
unit test at `lib/turnstile.test.ts`.

### Step 6 — `docs/security/auth-hardening.md` + cross-link

New file. Sections (table-of-contents in the doc itself):

1. **Threat model** — what attack classes we care about
   (credential stuffing, distributed brute-force, automated
   sign-up spam) and what's explicitly out of scope (DDoS at
   IP-flood scale, account takeover via session theft).
2. **Layered defense** — top-down enumeration of each layer +
   what it stops:
   - Vercel Edge — DDoS mitigation, free tier
   - per-IP rate limiter (M5, `lib/rate-limit.ts`) — single-IP
     credential stuffing, brute-force
   - per-email lockout (M6, `lib/rate-limit.ts`) — distributed
     credential stuffing against a known email
   - Cloudflare Turnstile CAPTCHA (M6) — automated abuse
     (sign-up spam, bot-driven credential stuffing)
   - Supabase Auth internal limits — generic abuse signals at
     the auth-provider layer
3. **Vercel WAF — deferred, Pro tier required** — explicit
   acknowledgment per spec acceptance criterion. One paragraph
   on what WAF would add (custom-rule firewall, geo-blocking,
   bot-management rules) and the criteria for revisiting
   (sustained abuse traffic visible in Sentry / admin
   notifications).
4. **Runbook — clearing a stuck email lockout** — exact CLI
   steps:
   - Open Upstash dashboard → Data Browser
   - Find the key `rl:email-lock:<email>` (e.g. `rl:email-lock
     :caregiver@example.com`)
   - Delete the key
   - Optionally also delete `rl:email-fails:<email>` to reset
     the in-window counter
   - The caregiver can now sign in immediately
5. **Runbook — interpreting rate-limit-breach notifications** —
   per-IP breach triage. Look at the IP, check whether it's
   geo-coherent with the legitimate caregiver, check recent
   Sentry events for context. Consider Vercel firewall IP block
   for sustained abusive IPs (Hobby tier supports manual IP
   blocking from the project dashboard even without Pro WAF).
6. **Runbook — interpreting email-lockout notifications** —
   per-email lockout triage. Treat as harassment / probing
   signal. The 15-minute cooldown is usually sufficient; manual
   clearance only when a known-legitimate caregiver reports
   being locked out and the lockout window hasn't expired.
7. **CAPTCHA accessibility fallback** — Turnstile's audio
   challenge, the support-email escalation path
   (`support@clearviewsavings.com`), and the documented manual
   workaround (admin temporarily disables the Supabase Auth
   CAPTCHA toggle during a Cloudflare outage).
8. **Env-var inventory** — `NEXT_PUBLIC_TURNSTILE_SITE_KEY`,
   `TURNSTILE_SECRET_KEY`, `TURNSTILE_TEST_BYPASS_KEY` + where
   each lives (Vercel Production / Preview / dev).

`docs/security/rls-audit.md` — add a one-line "see also" entry
at the top: `> See also: docs/security/auth-hardening.md for
the auth-surface protection model.` Reciprocal link in the new
doc.

### Step 7 — Production smoke + green-check close-out

Production smoke (manual, with user, during the Step 7 pause):

1. Visit `https://clearviewsavings.com/sign-up`. Turnstile
   widget renders. Without solving → submit → calm "Please
   verify and try again" message. Network tab confirms no call
   to Supabase Auth.
2. Solve the widget → submit with a throwaway email → land at
   `/check-your-email`. Confirmation email arrives. Verify
   email. Land at `/caregiver`.
3. Sign out. Visit `/sign-in`. Turnstile widget renders. Submit
   5 wrong passwords from one browser → see calm "Please wait
   a moment and try again" message on attempt 6 (per-IP
   limiter from M5).
4. Wait ~70 seconds for the per-IP window to clear. Open a
   second browser / incognito (or use a known-different IP via
   mobile data) and submit 5 wrong passwords against the same
   email — distributing across IPs. On attempt 6 (also from
   the second IP) see the calm error message. Confirm in
   Upstash dashboard that `rl:email-lock:<email>` exists.
   Confirm admin received the `email-lockout` notification at
   `ADMIN_EMAIL`.
5. Visit `/forgot-password`. Turnstile widget renders. Submit
   → calm "If that email exists, we sent a reset link" — reset
   email arrives. Click link → land at `/reset-password`
   (which intentionally has **no** widget). Set new password
   → land signed in.
6. Manually clear the lockout via the runbook. Sign in with
   the new password → land at `/caregiver`. ✓
