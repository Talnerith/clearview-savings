The spec's six open questions resolved at freeze:

1. **Email-template icon embedding** — external `<img
   src="https://clearviewsavings.com/branding/clearview-savings-icon.svg">`
   with the brand name in a typographic header as the always-visible
   fallback.

   `1.0`, **no session replay** (patient UI privacy posture).

   display name collected at first-patient-add, unchanged from M1.

   list, no DB role.
5. **Admin notifications** — four kinds: new caregiver verified,
   rate-limit breach, fatal Sentry event, daily caregiver-count
   digest at 09:00 UTC (Vercel Cron).
6. **E2E framework** — Playwright.

### Step 3 — Email confirmation flow + Supabase Auth Hook webhook

**Architecture (locked decision — replaces the SMTP approach in
the initial plan):** Supabase Auth's "Send Email Hook" fires for
every auth-related email event (signup, recovery, email change).
The hook POSTs a JSON payload to our webhook, signed with an HMAC
shared secret. Our route handler verifies the signature,
dispatches on `email_action_type`, renders the appropriate React
Email template from Step 2, and sends via Resend. Supabase Auth's
own email path is bypassed entirely — no SMTP wiring, no HTML
pasted into dashboard fields, templates remain in code.

**`/api/auth/email-hook` route handler** — new file
`app/api/auth/email-hook/route.ts`. POST endpoint.

```ts
import { Webhook } from "standardwebhooks";

export async function POST(req: Request) {
  const payload = await req.text();
  const headers = Object.fromEntries(req.headers);
  const wh = new Webhook(process.env.SUPABASE_AUTH_HOOK_SECRET!);
  let parsed: SupabaseEmailHookPayload;
  try {
    parsed = wh.verify(payload, headers) as SupabaseEmailHookPayload;
  } catch {
    return new Response("Invalid signature", { status: 401 });
  }

  const { user, email_data } = parsed;
  const { email_action_type, token_hash, redirect_to, site_url } = email_data;

  const confirmUrl =
    `${site_url}/auth/callback?token_hash=${token_hash}` +
    `&type=${email_action_type}` +
    `&next=${encodeURIComponent(redirect_to ?? "/caregiver")}`;

  switch (email_action_type) {
    case "signup":
      await sendEmail({
        to: user.email,
        subject: "Confirm your email at Clearview Savings",
        react: <ConfirmEmail confirmUrl={confirmUrl} />,
      });
      break;
    case "recovery":
      await sendEmail({
        to: user.email,
        subject: "Reset your Clearview Savings password",
        react: <ResetPassword resetUrl={confirmUrl} />,
      });
      break;
    // email_change handled if/when needed
    // magiclink + invite are out of scope (no magic links, open signup)
  }

  return Response.json({ ok: true });
}
```

**Install:** `pnpm add standardwebhooks`

**Payload shape** (from Supabase docs):

```ts
type SupabaseEmailHookPayload = {
  user: { id: string; email: string; /* ... */ };
  email_data: {
    token: string;
    token_hash: string;
    redirect_to: string;
    email_action_type: "signup" | "recovery" | "invite" | "magiclink" | "email_change";
    site_url: string;
    token_new?: string;
    token_hash_new?: string;
  };
};
```

**`/check-your-email` page** — new file
`app/(auth)/check-your-email/page.tsx`. Server component. Renders
calm copy ("Check your email for a confirmation link from Clearview
Savings"), a resend button (server action), a link to change email

**Resend confirmation action** —
`app/(auth)/check-your-email/actions.ts`. Calls
`supabase.auth.resend({ type: "signup", email })`. Same calm
inline message on success or failure ("If that email exists, we
sent a new confirmation. Please wait a moment before trying
again." — privacy-preserving phrasing). The resend re-triggers
the Auth Hook, which re-sends via our webhook.

**Sign-up redirect** — `app/(auth)/sign-up/actions.ts` currently
ends with `redirect("/check-your-email")` per M1; verify and
preserve. If sign-up currently auto-signs-in (because email
confirmation was disabled at some point in dev), the user's
Supabase dashboard flip in this step's pause will re-enable the
gate.

**Sign-in path** — verify that a caregiver whose email isn't yet
confirmed cannot reach `/caregiver`. Supabase Auth already
enforces this when "Confirm email" is on; verify the redirect
target is `/check-your-email`.

**Supabase Auth Hook setup (user does this in the dashboard
during the Step 3 pause):**

1. Open Supabase dashboard → your project → **Authentication** →
   **Hooks**.
2. Click **Add a new hook** → Type: **Send Email Hook**.
3. Hook URL: `https://clearviewsavings.com/api/auth/email-hook`.
   For dev testing, use the local Vercel preview URL or an
   ngrok-tunneled localhost URL.
4. Generate a secret (32+ random chars; `openssl rand -hex 32` or
   the dashboard's "Generate" button if present). Copy and save
   it — this becomes `SUPABASE_AUTH_HOOK_SECRET` in your env.
5. Click **Save**.
6. Open **Authentication** → **Providers** → **Email**. Ensure
   "Confirm email" toggle is **on**.
7. Send a test confirmation: sign up with a throwaway email,
   verify it arrives from `noreply@clearviewsavings.com` with the
   Clearview-Savings-branded template, click the link, verify it
   lands at `/caregiver`.

**Why this is better than the SMTP approach originally planned:**
- Templates live in code (React, type-checked, snapshot-tested)
- Single source of truth — no HTML pasted into dashboard fields,
  no `pnpm emails:build` step, no copy-paste rot
- Easy to iterate — change a React file, deploy, done

**Caveat — webhook downtime stops auth emails:**
- Mitigation: Sentry alerts on webhook 5xx (wired in Step 6's
  `fatal-sentry` admin notification path)
- Mitigation: the webhook is small and stateless; no DB writes
- Mitigation: Vercel's regional redundancy + Supabase's webhook
  retry-with-backoff cover transient failures

### Step 6 — Admin notifications + `/admin` gate

**`lib/admin-email.ts`:**

```ts
type AdminNotificationKind =
  | { kind: "new-caregiver"; caregiverEmail: string;
      caregiverId: string }
  | { kind: "rate-limit-breach"; ip: string; endpoint: string;
      attempts: number }
  | { kind: "fatal-sentry"; eventId: string; title: string;
      url: string }
  | { kind: "daily-digest"; caregiverCount: number;
      patientCount: number; activeLast7d: number };

async function sendAdminNotification(
  notification: AdminNotificationKind
): Promise<void>
```

Calls `sendEmail` (Step 2) with `to: process.env.ADMIN_EMAIL` and
the `<AdminNotification>` React Email template. Silent no-op if
`ADMIN_EMAIL` is unset (dev convenience — don't crash if env
isn't configured locally).

**Wiring:**

- **new-caregiver** fires from `getCurrentCaregiver()` (the helper
  that auto-creates the `caregivers` row on first sign-in)
  immediately after the insert succeeds. One-shot per caregiver.
- **rate-limit-breach** fires from `lib/rate-limit.ts` (Step 7)
  when an IP exceeds the threshold by N× (e.g., 3× the threshold —
  routine block-on-first-violation would spam admin email).
- **fatal-sentry** wired via a Sentry webhook to a Next.js route
  handler at `app/api/sentry-webhook/route.ts`. The webhook URL is
  registered in Sentry's project settings and signed with a
  shared-secret env var (`SENTRY_WEBHOOK_SECRET`).
- **daily-digest** wired via Vercel Cron. `vercel.json` configures
  `crons: [{ "path": "/api/cron/daily-digest", "schedule": "0 9 *
  * *" }]` (09:00 UTC). The cron route handler queries counts and
  calls `sendAdminNotification`. The route verifies
  `Authorization: Bearer ${process.env.CRON_SECRET}` per Vercel's
  cron convention to prevent unauthenticated triggers.

**`/admin/*` route gating:**

`middleware.ts` (or a per-page check on the admin layout):

```ts
const { data: { user } } = await supabase.auth.getUser();
if (user?.email !== process.env.ADMIN_EMAIL) {
  return new Response("Not found", { status: 404 });
}
```

404 (not 403) to avoid revealing the admin route exists.

**M5 admin routes:** with the waitlist pivot, the only admin
surface is informational — `/admin/page.tsx` shows recent
caregivers, recent Sentry events count, recent rate-limit breaches
(in-memory rolling window via Upstash). Lightweight — admin UX
isn't M5's core, just enough to confirm ops health.

4. Paste both into Vercel project env vars (production) and

**`lib/rate-limit.ts`:**

```ts
const limiters = {
  signUp: new Ratelimit({ redis, limiter: Ratelimit.slidingWindow(5, "1 m") }),
  signIn: new Ratelimit({ redis, limiter: Ratelimit.slidingWindow(5, "1 m") }),
  forgotPassword: new Ratelimit({ redis, limiter: Ratelimit.slidingWindow(5, "1 m") }),
};

async function checkRateLimit(
  identifier: string,
  action: keyof typeof limiters
): Promise<{ allowed: boolean; remaining: number }>
```

Bypass in non-production via:
```ts
if (process.env.NODE_ENV !== "production") return { allowed: true, remaining: 999 };
```

**Wiring:** each auth server action calls `checkRateLimit(ip,
"signUp" | "signIn" | "forgotPassword")` before doing the actual
auth call. On `!allowed`, redirect with `?error=rate_limited`. The
auth pages render the calm "Please wait a moment and try again."
inline message for that error code.

**IP source:** `headers().get("x-forwarded-for")?.split(",")[0]
?? "unknown"`. Vercel populates this header in production. In
dev the `unknown` fallback never matters because the bypass is
active.

**Tests:** `lib/rate-limit.test.ts` — mocked `@upstash/ratelimit`,
verify the threshold + the calm-message error path. Test the
production-only gate. Real Upstash calls aren't exercised in CI.

### Step 8 — RLS audit + cross-tenant integration test

**Audit walkthrough** — for every table in `lib/db/schema.ts`,
verify policies in `supabase/policies.sql`:

- `caregivers` — caregivers can only select/update their own row
  (`id = current_caregiver_id()`)
- `patients` — caregivers can CRUD only their own patients
  (`caregiver_id = current_caregiver_id()`)
- `accounts` — scoped via `patients` join; SELECT/INSERT/UPDATE
  scoped, DELETE restricted to caregiver-owned
- `transactions` — scoped via `accounts → patients`
- `scheduled_deposits` — scoped via `accounts → patients`
- `deposit_codes` — scoped via `patients`
- `audit_log` — scoped to `caregiver_id = current_caregiver_id()`

For each policy, record in `docs/security/rls-audit.md`:
- Table name
- Policy name (SELECT / INSERT / UPDATE / DELETE)
- Scope clause
- Verdict (✓ correct / ✗ needs fix)
- If ✗: the fix, applied in a new `supabase/policies.sql`
  revision

**Service-role audit:** grep `lib/supabase` and route handlers for
any use of the service-role key. Confirmed allowed call sites:
- `scripts/seed.ts` (dev only, never reached by request handler)
- `getCurrentCaregiver()` first-insert path (uses service role to
  bypass RLS for the bootstrap insert — verify this is still the
  case and that the auth check is solid)
- M1 patient route's privileged DB (bypasses RLS for the
  unauthenticated patient render — verify the patient-UUID
  capability is the only gate)

Document each in the audit doc.

**Integration test:** `lib/security/rls-cross-tenant.test.ts`.
Against pg-mem with the production policies loaded:

1. Create caregivers A and B
2. Create one patient for each (Pat_A, Pat_B)
3. Create one account for each
4. Create one transaction, scheduled deposit, deposit code, and
   audit log entry for each
5. For each of the 7 tables, in caregiver A's session, attempt:
   - SELECT B's row by id → assert empty result
   - UPDATE B's row by id → assert 0 rows affected
   - DELETE B's row by id → assert 0 rows affected
   - INSERT a row tying to B's patient/account → assert error or 0
     rows
6. Repeat from B's session against A's data (defense against
   asymmetric policies)

Test count: roughly 7 tables × 4 ops × 2 directions = 56
assertions (some ops may not apply per table).

**Caveat:** pg-mem's RLS support is partial — it doesn't honor
PostgreSQL RLS policies natively. The integration test runs in
"emulated RLS" mode: helper functions wrap each query in a
`SET app.current_caregiver_id = ...` and the queries themselves
include the scope check explicitly. This catches policy
correctness at the SQL-shape level but doesn't prove the actual
RLS engine enforces them.

**Real-Postgres cross-check (locked decision — testcontainers):**

A second test file `lib/security/rls-cross-tenant.real.test.ts`
runs the same matrix against a real Postgres container.

```ts
import { PostgreSqlContainer } from "@testcontainers/postgresql";

describe.skipIf(!process.env.RUN_REAL_POSTGRES_TESTS)(
  "RLS cross-tenant (real Postgres)",
  () => {
    let container: StartedPostgreSqlContainer;
    let url: string;

    beforeAll(async () => {
      container = await new PostgreSqlContainer("postgres:15").start();
      url = container.getConnectionUri();
      // 1. Run drizzle migrations against `url`
      // 2. Load supabase/policies.sql against `url`
      // 3. Install the auth.uid() stub (see below)
    }, 60_000);

    afterAll(async () => {
      await container?.stop();
    });

    // ... matrix tests, same shape as the emulated test but
    // using a real-Postgres connection and `SET LOCAL request.
    // jwt.claim.sub = '<caregiver-uuid>'` to switch identities
  }
);
```

**Install:** `pnpm add -D @testcontainers/postgresql testcontainers`

**Supabase `auth.uid()` stub** — Supabase's RLS policies use
`auth.uid()` to read the signed-in user's id. Real Postgres
doesn't have Supabase's `auth` schema. We stub it as a function
reading from a session variable, applied as a SQL fixture before
the policies load:

```sql
create schema if not exists auth;
create or replace function auth.uid() returns uuid as $$
  select coalesce(
    current_setting('request.jwt.claim.sub', true),
    ''
  )::uuid
$$ language sql stable;
```

Then each test switches identity with
`SET LOCAL request.jwt.claim.sub = '<caregiver-uuid>'` inside a
transaction.

**Gating:** `RUN_REAL_POSTGRES_TESTS=1 pnpm test --run` runs
both the emulated test and the real-Postgres test. Default
`pnpm test --run` runs only the emulated test, so CI without
Docker still passes. Local dev with Docker Desktop on Windows
satisfies the container requirement.

**Cost:** ~5–10s container boot per test run; acceptable
because it's opt-in. Run before Step 9 deploy and as part of
the M5 close-out green-check.
