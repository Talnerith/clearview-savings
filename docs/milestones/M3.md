# Milestone 3 — Workbooks (printable activity sets + single-use reward codes)

Builds on `m2-complete`. The shape of M3 is a near-mirror of M2: the caregiver
generates a printable artifact carrying a single-use code; the patient
completes the work, enters the code on their bank screen, and the reward
amount posts to their account exactly once. The difference from M2 is the
artifact — instead of a single check, it's a multi-page workbook of math /
reading / sequencing / logic exercises tuned to be doable but engaging for
someone with mild-to-moderate dementia.

The same `deposit_codes` table carries both check and workbook codes; the
atomic redemption path (`redeemCode` in `lib/deposit-codes.ts`) is shared.
Workbook-specific columns are added to `deposit_codes` rather than forking
into a parallel `workbook_codes` table.

## Build plan — 6 steps

| #   | Step                                                  |
| --- | ----------------------------------------------------- |
| 1   | Schema + migration + workbook columns on deposit_codes |
| 2   | Static problem bank + deterministic sampler           |
| 3   | Caregiver workbook generator (list, form, PDF, answer key) |
| 4   | Patient submit-work flow (code → confirm → done)      |
| 5   | Tests (race, sampler determinism, bank sanity)        |
| 6   | Branding + patient-vocabulary audit                   |

Pause points: end of Step 3 (caregiver flow wired up before patient flow);
end of Step 4 (patient flow wired up before tests/audit).

## Scope and decisions

### Data model — extend, don't fork

`deposit_codes.kind` already had `'workbook'` as a value from M1. M3 adds
three nullable columns scoped to that kind:

- `workbook_kind` (enum `math | reading | mixed`) — caregiver-facing category.
- `workbook_grade` (smallint, 2 or 3) — difficulty band.
- `content_seed` (jsonb) — a frozen snapshot of the exact problems printed on
  the workbook, so the caregiver answer-key view always matches the printed
  copy even if the upstream problem bank changes later.

Application layer enforces the "non-null iff `kind = 'workbook'`" invariant,
same pattern `memo` already uses (check-only, null for workbooks). No CHECK
constraint added; M2 doesn't have one for `memo` either.

`label` doubles as the workbook title. Auto-generated default is
`Activity Set #N` where N is the count of prior workbook codes for the
patient + 1. Label becomes the transaction's display label on Recent
Transactions when the code is redeemed — must read like a real bank line item.

### Typed-amount step is dropped for workbooks

M2 keeps a "type the amount" step in the check deposit wizard as
therapeutic theatre — it preserves the deposit-at-the-teller ritual.
Workbooks have no equivalent real-world ritual; the code IS the proof of
completed work. Patient flow is two steps:

1. Code entry
2. Confirmation card (workbook title + reward amount) with a Submit button
3. Done page (mirrors check flow)

### Content bank: static JSON, AI-ready seam

500 hand-authored problems across 10 JSON files (2 grades × 5 categories ×
50 each):

- `math-facts` — single-line arithmetic (add/sub/mul/div within grade band)
- `word-problems` — short real-world adult scenarios (groceries, gardening,
  walks, weather)
- `reading-passages` — 2–4 sentence passages, 2–3 comprehension questions
- `sequencing` — 4–5 life-task steps to put in order
- `simple-logic` — odd-one-out, next-in-pattern, simple comparison, if/then

Authoring rules: real-world adult contexts only. No fairy tales, no cartoon
characters, no schoolroom framing. Numbers and words scaled per grade band.

The bank lives behind a single resolver `getProblemBank(grade, category)` in
`lib/workbook-content/index.ts`. **This is the AI-ready seam.** Swapping it
for an async fetch-and-cache layer (Anthropic API or otherwise) keeps the
sampler, PDF renderer, and `content_seed` snapshot pipeline unchanged.
Generation-mode is **out of scope for M3** — the seam exists, the
implementation does not.

### Sampler — deterministic, seeded, idempotent

`sampleWorkbook({ kind, grade, seed })`:

- xmur3 (string → 32-bit hash) feeds mulberry32 (deterministic RNG)
- Seeded Fisher-Yates shuffle
- Walks a `PAGE_SHAPES` map (5 pages per workbook, density tuned per kind)
- Shuffles each needed category exactly once; advances per-category cursor
  so no problem repeats within a workbook even when a category repeats
  across pages
- `newWorkbookSeed()` returns a 16-byte hex string via
  `crypto.getRandomValues`, stored on the row for traceability

`PAGE_SHAPES`:

- `math`: 8 + 3 + 8 + 3 + 6 (math-facts/word-problems mix)
- `reading`: 1 + 2 + 1 + 4 + 1 (passages / sequencing / logic mix)
- `mixed`: 6 + 1 + 2 + 3 + 2 (across all five categories)

### Sequencing — canonical order in the bank, scrambled at render

`SequencingProblem.correctSequence` stores the canonical order. The PDF
renderer scrambles for display using a stable derived seed
(`${workbookSeed}:${problem.id}`) so the printed scrambled order is
reproducible; the answer-key renderer prints the canonical order. This avoids
carrying a pre-shuffled `items` array and indirection map in JSON.

### Patient vocabulary — non-negotiable

These tokens must be **zero matches** under `app/(patient)/**` after Step 6:

- `\bworkbook\b`, `\bworksheet\b`, `\bgrade\b`, `\bexercise\b`,
  `\bactivity book\b`

Acceptable patient-facing terms:

- "Submit Completed Work" — primary CTA, page heading
- "Activity Set #N" — the auto-generated workbook title (becomes the
  transaction label)
- Caregiver-set custom titles (the input is theirs to choose)

PDF rendering uses neutral page-category headers: "Number practice",
"Word problems", "Reading", "Putting things in order", "Thinking puzzles" —
never the internal category enum values.

### Single-error error model (same as M2)

Any redemption failure path — bad UUID, malformed code, code not found,
code already used, `redeemCode` throws — redirects back to the code-entry
step with the same `?error=invalid_or_used`. Single inline message
("We couldn't read that code. Please check the code or ask for help.").
Don't add per-reason messages; small but defensible privacy choice.

### PDF rendering — Times-Roman in @react-pdf/renderer

The base font set covers ASCII + Latin-1 only. Glyphs outside that range
render as **blank** (silently). Math banks use U+2212 MINUS SIGN as the
typographically correct subtraction operator; the PDF render path
substitutes `−` → `-` and `—` → ` - ` at draw time via a `pdfSafe()` helper.
The bank source stays typographically canonical so the HTML answer-key page
keeps the prettier glyph. If a future bank entry introduces another
non-Latin-1 glyph, extend `pdfSafe()` or register a Unicode-coverage font.

## What's deliberately NOT in scope

- **AI-mode generation.** The seam exists at `getProblemBank`; the
  implementation lands behind a feature flag in a later milestone.
- **Per-patient daily/weekly deposit limits.** Carried over from M2.
- **Voiding a code.** `deposit_codes.status` enum is still `unused | used`;
  no `voided` state added in M3.
- **Multi-account routing.** Workbook reward still lands on the patient's
  first-created account (same as M2 checks).
- **Real image / handwriting capture or grading.** The submit flow accepts
  the code only — no scanning, no OCR, no answer validation.
- **Audit log table.** Still deferred.
- **Custom workbook PDFs (logos, letterhead).** Cover page is plain
  brand-name text.

1. Read `CLAUDE.md` (project rules)

3. Read this file — `docs/milestones/M3.md` (stable plan)
4. Read `docs/milestones/M3-progress.md` (current state, snapshot in time)
5. Inspect actual files to confirm the progress doc matches reality
6. Run `pnpm typecheck && pnpm lint && pnpm test --run` for a green baseline
7. Wait for "go" before writing code
